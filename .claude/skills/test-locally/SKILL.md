# 本地测试

修改任何 skill / hook / command 后，按以下步骤在本地测试：

1. **选测试范围**——输入：本次改动涉及的插件数；输出：加载方式。只涉及一个插件就用单插件加载（更快、避免其它插件干扰），涉及跨插件联动就全量加载。
2. **在仓库根目录执行对应命令**——输入：上一步选定的加载方式；输出：一个已加载对应插件的新会话。

   ```bash
   # 全量加载（推荐用于完整测试/跨插件联动）
   claude --plugin-dir .

   # 单插件加载
   claude --plugin-dir ./plugins/optimus-devops-plugin
   ```

3. **在新会话里直接输入触发词验证行为**——输入：上一步的新会话；输出：改动是否符合预期。无需重启、无需重新安装，文件改动立即生效。

## 故障排查

| 触发条件 | 一线修复 | 仍失败兜底 |
|---|---|---|
| 报错找不到插件/目录 | 确认当前在仓库根目录执行（`--plugin-dir .` 依赖相对路径），用 `pwd` 确认所在目录 | 改用绝对路径重试；若仍报错，检查目标目录下是否有 `skills/`、`hooks/` 等标准插件结构 |
| 输入触发词没反应 | 确认改的 skill 是否在这次 `--plugin-dir` 加载到的插件目录内（单插件加载模式下其余插件不会被注册） | 换成全量加载 `claude --plugin-dir .` 重新验证 |
| 改了 SKILL.md 但行为没变 | 确认是新开的会话（同一会话内 skill 定义不会热更新），退出重进一个新会话 | 用 `/plugin` 查看当前会话实际加载了哪些 skill，排查 frontmatter 是否有 YAML 语法错误导致该 skill 整体未被解析 |

## 不要做

- 不要把绝对路径写死进文档或脚本再提交——不同机器盘符、路径不同会直接失效（本 skill 已改用相对路径，别再改回绝对路径）
- 其余典型误用（不在仓库根目录用相对路径、改完在旧会话原地验证、混淆多会话测试范围）已在上方"故障排查"表对应行给出修复方式，此处不重复列出
