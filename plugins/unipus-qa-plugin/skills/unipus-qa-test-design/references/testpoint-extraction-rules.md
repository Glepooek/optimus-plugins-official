# 测试点提取规则

## 规则1：功能点识别与分组

按业务逻辑识别主要功能点，再在每个功能点下细分验证类型：

- **功能点划分**：按业务逻辑拆分为独立的功能模块
- **层级结构**：模块 → 功能点 → 验证类型 → 具体测试点
- **功能聚合**：将相关操作归类到同一功能点下

## 规则2：功能动作识别

从需求中识别核心功能动作，每个动作对应一个测试点：

- **CRUD操作**：创建、读取、更新、删除
- **业务动作**：登录、注册、支付、审核、发布等
- **状态变更**：启用/禁用、上线/下线、通过/拒绝等

## 规则3：业务规则提取

将需求中的业务规则转化为验证点：

- **条件判断**："当满足X条件时，执行Y操作" → 对应功能验证点
- **约束限制**："用户名长度不超过20个字符" → 对应边界验证点
- **权限控制**："只有管理员可以删除数据" → 对应功能/异常验证点

## 规则4：数据验证点识别

识别需要验证的数据相关要点：

- **数据格式验证**：邮箱格式、手机号格式
- **数据完整性**：必填字段、关联数据
- **数据一致性**：跨模块数据同步

## 规则5：异常场景覆盖

从需求中推导可能的异常情况：

- **输入异常**：空值、超长、特殊字符
- **系统异常**：网络中断、服务不可用
- **权限异常**：无权限访问、会话过期

## 优先级设置指南

| 优先级 | 适用场景 |
|--------|----------|
| **高** | 核心业务功能、安全相关、数据完整性 |
| **中** | 边界条件、非核心功能、格式验证 |
| **低** | 边缘异常场景、UI细节、提示文案 |

## 测试点粒度原则

- **原子性**：一个测试点只验证一个要素
- **独立性**：测试点之间避免重复
- **可测性**：验证要点明确具体，可执行验证
- **适中性**：不过细（拆成测试用例级别），不过粗（包含多个不相关要素）
