# 测试计划各章节详细生成规则

## 1. 项目概述

### 1.1 项目背景

**数据来源**：需求拆分文档的文档标题和第一个功能模块的功能概述

**生成规则**：
- 第一句：说明产品是什么（从文档标题提取产品名称）
- 第二句：说明产品解决什么问题（从功能概述中提炼核心价值）
- 第三句：说明产品的核心能力（列出主要功能模块的关键词）

**示例**：
```
智课数字人视频生成平台是一个基于AI技术的在线教育视频制作平台。
平台通过数字人形象和语音合成技术，支持用户将PPT课件自动转化为
带有数字人讲解的教学视频，降低视频课程制作门槛，提升内容生产效率。
核心功能包括数字人形象管理、课程素材上传、课件内容编辑、视频生成
与导出、历史记录管理等。
```

### 1.2 系统功能概述

**数据来源**：需求拆分文档的所有功能模块

**生成规则**：
- 按编号列出所有功能模块
- 每个模块用一句话描述核心功能（取自该模块的"功能概述"段落）
- 如有架构文档，在模块列表后补充核心业务流程图或流程描述

**格式**：
```markdown
系统主要包含以下N个核心功能模块：
1. **[模块名称]**：[一句话功能描述]
2. **[模块名称]**：[一句话功能描述]
...
```

### 1.3 测试目标

**生成规则**：固定框架 + 从NFR中提取量化指标

**固定目标**：
- 验证系统各功能模块的正确性和完整性
- 确保数据处理的准确性和可靠性
- 验证用户体验和界面友好性
- 确保数据安全性和系统性能
- 验证多端兼容性

**量化目标提取**：
- 扫描NFR中的性能指标 → 添加"系统响应时间满足NFR-XXX要求"
- 扫描NFR中的质量指标 → 添加具体准确率目标
- 扫描NFR中的可用性指标 → 添加可用性目标

## 2. 测试范围

### 2.1 In Scope 生成规则

**数据来源**：需求拆分文档的所有功能模块 + 架构文档的接口清单

**生成规则**：
1. 遍历需求拆分文档中的每个功能模块
2. 对每个模块，提取其子功能分组名称和关键FR条目
3. 格式化为带粗体模块名 + 冒号 + 关键功能点列表

**格式**：
```markdown
- **[模块名称]**：[关键功能点1]、[关键功能点2]、[关键功能点3]
```

**补充规则**：
- 如有架构文档，额外添加"接口测试"范围，列出各模块的接口数量
- 如需求中包含埋点/监控类功能，单独列出
- 如需求中包含第三方集成（如SSO），列出集成测试范围

### 2.2 Out of Scope 生成规则

**固定排除项**：
- 第三方服务内部实现（从架构文档的第三方依赖中提取具体服务名）
- 服务器硬件性能测试
- 网络基础设施测试
- 操作系统级别安全测试

**动态排除项**：
- 如架构文档中有SSO/中台集成 → 排除"中台系统内部功能"
- 如NFR兼容性只列出主流浏览器 → 排除"非核心浏览器兼容性"

## 3. 测试策略

### 3.1 测试类型生成规则

**自动判断逻辑**：

```
IF 有功能模块 → 添加"功能测试"
IF 有UI相关FR → 添加"用户界面测试"
IF 架构文档有接口清单 → 添加"接口测试"
IF NFR有性能指标 → 添加"性能测试"
IF NFR有安全要求 → 添加"安全测试"
IF NFR有兼容性要求 → 添加"兼容性测试"
IF 架构文档有第三方依赖 → 添加"集成测试"
IF 有AI相关功能 → 添加"AI功能测试"（准确性、鲁棒性、一致性）
IF 有多用户协作功能 → 添加"并发测试"
ALWAYS → 添加"回归测试"
```

**每种测试类型的描述模板**：
```markdown
- **[测试类型]**：[一句话描述验证目标]
```

### 3.2 测试方法

固定三种方法，根据项目特点调整描述：
- 手动测试：功能验证、UI验证、探索性测试
- 自动化测试：核心流程回归、接口自动化
- 探索性测试：发现边界场景和潜在问题

### 3.3 测试优先级

**P1（高优先级）判定规则**：
- AC中标注了量化指标的功能
- 涉及数据安全和用户认证的功能
- 核心业务流程（从架构文档的业务流程图中识别）
- 涉及资金/支付的功能

**P2（中优先级）判定规则**：
- UI界面友好性
- 文件格式兼容性
- 非核心辅助功能
- 报告生成和导出

**P3（低优先级）判定规则**：
- 界面美化细节
- 高级配置选项
- 边缘场景

## 4. 测试环境

### 4.1 硬件环境

**服务器配置推导规则**：
- NFR并发数 ≤ 50 → 4核CPU、8GB内存
- NFR并发数 50-200 → 8核CPU、16GB内存
- NFR并发数 > 200 → 16核CPU、32GB内存
- 磁盘：有大文件上传功能 → 500GB+，否则 256GB

**客户端配置**：固定为主流配置（4核CPU、8GB内存、256GB磁盘）

### 4.2 软件环境

**操作系统**：从NFR兼容性要求中提取，默认 Windows 10/11、macOS 12+
**浏览器**：从NFR兼容性要求中提取，默认 Chrome 90+、Firefox 88+、Safari 14+、Edge 90+
**数据库**：从架构文档中提取
**网络**：有线100Mbps + 无线50Mbps

### 4.3 测试数据

**生成规则**：按以下七类系统性扫描需求文档和架构文档，输出结构化测试数据准备清单，并在末尾生成数据准备 Checklist。

---

#### 类型一：用户账号数据

**不只按角色分类，要按"业务状态 × 场景"细分账号，每个账号标明角色、预置状态和专用测试场景。**

```
FOR EACH 用户角色（从FR/NFR中提取）:
  生成账号条目：账号标识、角色、状态描述、专用测试场景

场景分类规则：
  - 全新账号（无历史数据）：测试初始状态展示、空列表
  - 有完整历史数据账号：测试列表、分页、汇总数据
  - 有中途未完成任务的账号：测试断点恢复场景
  - 受限/未授权账号：测试权限拦截
  - 管理员账号：测试角色隔离
  - 未登录（无Session）：测试未登录跳转

每个账号条目格式：
  | 账号标识 | 角色 | 状态描述 | 用途 |

注意：密码统一规则须写入账号说明，测试用例引用时不硬编码密码。
```

---

#### 类型二：测试文件

```
FOR EACH FR条目:
  IF 包含"上传"、"导入"、"附件" → 提取：
    - 支持格式（列出所有格式，分正常格式和需测试的异常格式）
    - 正常大小文件（符合要求，如 1MB PDF）
    - 边界大小文件（接近上限，如 49.9MB 和超限 51MB）
    - 异常格式文件（伪装成合法格式的错误文件，如 .exe 改名为 .pdf）

IF 含音频/视频功能（录音、播放、媒体上传）→ 额外生成：
  - 各边界时长的音频/视频文件（如 0.5s、临界最大值、超限值）
  - 标准媒体文件（用于对比播放、标准音源）
  - 损坏/空内容的媒体文件（测试异常处理）
  - 文件命名规则说明（如 {id}_standard.mp3）
  - 文件存储位置（CDN/OSS 路径说明）
```

---

#### 类型三：业务基础数据

```
FOR EACH 功能模块:
  IF 模块依赖列表/分类/模板等前置数据 → 列出预置数据需求：
    - 数据类型（如课程分类、题库、模板、单词列表等）
    - 数量要求（功能测试用少量，性能测试用大量，说明两个场景）
    - 关键字段值（列出代表性的数据样本字段，覆盖不同等级/分类）

IF 含评分/分级功能 → 必须覆盖所有评分等级/分级边界：
  - 每个等级至少准备1条代表性数据（含上下边界值）
  - 示例：优秀（90-100）、良好（75-89）、及格（60-74）、需改进（0-59）

IF 含历史记录/列表功能 → 预置数据量需满足分页测试：
  - 数量 = 每页条数 + 1（触发第二页）
  - 数据时间戳需覆盖"倒序排列"验证（最新记录应排第一）
  - 汇总字段（总次数、最高分、平均分）需人工计算后写入测试断言
```

---

#### 类型四：Mock / Stub 数据

```
IF 有第三方服务（AI评分、支付、短信等）:
  FOR EACH 服务:
    生成 Mock 场景清单，覆盖：
      - 正常成功场景（含所有返回等级/状态的变体）
      - 超时场景（模拟响应时间超过NFR阈值）
      - 错误/失败场景（HTTP 4xx/5xx 响应）
      - 触发方式（请求头标志位 / 特定入参 / 配置开关）

    输出格式：
    | Mock 场景 | 触发条件 | 返回数据 |

    标注哪些测试用例依赖"真实服务"（非Mock），需在联调阶段单独验证。
```

---

#### 类型五：边界与专项数据

```
FOR EACH FR中有输入字段的条目:
  生成边界数据清单：
    - 最大长度输入（如名称字段最大字符数）
    - 最小长度/空值
    - 特殊字符（< > & " ' SQL注入、XSS特殊字符）
    - 重复值（用于测试唯一性校验）
    - 格式错误（如邮箱、手机号格式非法）

IF 有限流/防刷功能（如NFR：N秒内最多M次）:
  生成限流专项数据表：
    | 操作次数 | 操作间隔 | 预期结果 |
    覆盖：第1次、临界次（第M次）、超限次（第M+1次）、
          间隔超时后重置（间隔>N秒后第1次）

IF 有时长/大小限制:
  生成边界值数据：0、1、临界值-1、临界值、临界值+1
```

---

#### 类型六：性能测试数据

```
IF NFR中有性能/并发指标:
  列出性能数据需求：
    - 数据量级（如模拟N个用户、M条记录）
    - 生成方式（SQL脚本/工具脚本/API批量写入）
    - 数据特征（均匀分布/集中峰值）
```

---

#### 类型七：数据准备 Checklist

**必须在章节末尾生成 Checklist 表格，列出所有数据准备项、责任人和状态。**

```
FOR EACH 数据准备项（汇总以上六类）:
  生成 Checklist 行：
    | 类别 | 准备项 | 负责人 | 状态 |
    状态默认填 ☐（待办）

必须包含：
  - 账号创建项（每类账号 1 行）
  - 业务数据/单词/课程创建项
  - 文件/音频上传项（含CDN/存储路径确认）
  - Mock 服务部署验证项
  - 历史记录预置项（含预置数量）
  - 测试工具配置项（弱网、抓包工具等）
  - 每轮测试前数据清理项（注明哪些账号需重置）
```

### 4.4 网络环境

- 如NFR有网络相关要求 → 按要求列出
- 默认包含：内网环境、外网环境、弱网环境

## 5. 测试用例设计

### 5.1 用例数量估算规则

**按模块估算**：

```
模块预估用例数 = FR条目数 × 系数

系数规则：
- 包含文件上传功能的模块：× 3（正常格式 + 异常格式 + 大小边界）
- 包含CRUD操作的模块：× 2.5（增删改查 + 异常）
- 纯展示类模块：× 1.5（正常展示 + 兼容性）
- 包含AI功能的模块：× 3（准确性 + 鲁棒性 + 边界）
```

**接口用例估算**（如有架构文档）：
```
接口用例数 = 接口数量 × 3（正常 + 异常参数 + 权限）
```

**集成用例**：
```
集成用例数 = 核心业务流程数 × 2（正常流程 + 异常中断）
```

### 5.1 输出格式

```markdown
| 功能模块 | FR条目数 | 预估用例数量 | 覆盖的验证点 |
|----------|---------|-------------|-------------|
| [模块名] | [数量]  | [估算值]    | [关键验证点] |
```

末尾添加汇总行：总FR条目数、总预估用例数

## 6. 测试执行计划

### 6.1 阶段划分规则

**小型项目**（FR ≤ 50，模块 ≤ 5）：
- 准备1天 → 功能测试3天 → 集成/性能1天 → 兼容性1天 → 回归1天 → 总结1天
- 总计约8天

**中型项目**（FR 50-150，模块 5-10）：
- 准备2-3天 → 功能测试5-10天 → 集成/性能3天 → 兼容性2天 → 回归2天 → 总结1-2天
- 总计约15-22天

**大型项目**（FR > 150，模块 > 10）：
- 准备3-5天 → 功能测试10-20天 → 集成/性能5天 → 安全3天 → 兼容性3天 → UAT5天 → 回归3天 → 总结2天
- 总计约34-46天

### 6.2 详细时间安排表格式

```markdown
| 阶段 | 活动 | 开始时间 | 结束时间 | 工作量(人天) | 负责人 |
|------|------|----------|----------|-------------|--------|
| 准备阶段 | 测试环境搭建 | D1 | D2 | 2 | 测试工程师 |
| 功能测试 | [模块名]测试 | D3 | D5 | 3 | 测试工程师A |
| ... | ... | ... | ... | ... | ... |
```

**时间安排规则**：
- 使用相对日期（D1、D2...），不使用绝对日期
- 可并行的模块安排并行执行
- 集成测试在功能测试完成后开始
- 回归测试在缺陷修复后开始
- 每个阶段之间预留0.5-1天缓冲

## 7. 测试资源

### 人员配置推算规则

```
总工作量(人天) ÷ 项目周期(天) = 最少测试人员数（向上取整）

角色分配：
- 测试负责人：1人（固定）
- 功能测试工程师：max(1, 总工作量÷项目周期 - 1) 人
- 自动化测试工程师：如有自动化需求 → 1人
- 性能测试工程师：如有性能测试 → 1人
- 安全测试工程师：如有安全测试 → 1人
```

## 8. 风险评估

### 风险提取规则

**从架构文档提取技术风险**：
```
FOR EACH 第三方服务:
  生成风险："[服务名]服务不稳定"，概率=中，影响=高
  缓解措施："建立监控机制，设置超时和重试"

FOR EACH 文件上传功能:
  生成风险："大文件处理失败"，概率=中，影响=中
  缓解措施："优化上传机制，添加断点续传"

IF 有AI功能:
  生成风险："AI处理准确性不达标"，概率=中，影响=高
  缓解措施："准备多样化测试数据，建立准确率基准"

IF 有异步任务:
  生成风险："异步任务处理超时"，概率=中，影响=中
  缓解措施："设置合理超时阈值，添加进度监控"
```

**固定项目风险**：
- 时间风险：测试时间因缺陷修复延长 → 预留缓冲时间
- 资源风险：测试人员技能不足 → 提供培训
- 环境风险：测试环境不稳定 → 准备备用环境
- 需求风险：需求变更影响计划 → 建立变更控制流程

## 9. 准入/准出标准

### 准入标准（固定）
- 代码已完成并部署到测试环境
- 测试环境已搭建完成
- 测试数据已准备就绪
- 测试用例已设计完成
- 开发自测通过

### 准出标准

**固定标准**：
- 高优先级用例通过率 100%
- 中优先级用例通过率 ≥ 90%
- 低优先级用例通过率 ≥ 80%
- 所有严重/关键缺陷已修复

**动态标准**（从NFR提取）：
```
FOR EACH NFR性能指标:
  添加："[指标名]满足NFR-XXX要求（[具体值]）"

FOR EACH NFR安全要求:
  添加："安全测试无高危漏洞"

FOR EACH NFR兼容性要求:
  添加："主流浏览器兼容性测试通过"
```
