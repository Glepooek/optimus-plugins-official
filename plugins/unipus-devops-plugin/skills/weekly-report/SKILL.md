# 工作周报转写技能

将零散的工作内容转写为结构清晰、量化成果的专业周报。

## 工作流程

### 第一步：收集工作内容

从以下来源获取本周工作信息（可组合使用）：

**来源 1：对话描述**
- 如果用户已描述工作内容，从对话历史中提取
- 如果用户只说"帮我写周报"而未提供内容，主动询问："请简要描述本周的主要工作内容，包括完成的项目、修复的问题、参与的会议等"

**来源 2：工作日志文件**
如果用户提供了日志文件（如 daily-log.md、工作记录.txt），读取并提取关键工作项。

**来源 3：Git 提交记录（推荐）**

自动提取本周的 git 提交记录作为工作素材。使用以下跨平台兼容的命令：

```bash
# 方案 1：计算本周一日期（推荐，兼容所有平台）
# 由 Claude 计算本周一的日期，替换 YYYY-MM-DD
git log --since="YYYY-MM-DD" --until="now" --oneline --author="$(git config user.name)"

# 方案 2：使用相对天数（通用但范围稍大）
git log --since="7 days ago" --oneline --author="$(git config user.name)"

# 获取更详细的提交信息（含文件变更）
git log --since="YYYY-MM-DD" --until="now" --stat --author="$(git config user.name)"
```

**边界情况处理：**
- 如果当前目录不是 git 仓库，提示用户："当前目录不是 git 仓库，请切换到项目目录或跳过 git 提取"
- 如果本周无提交记录，提示："本周未发现 git 提交记录，请通过对话补充工作内容"
- 如果用户指定了仓库路径，使用 `git -C <path>` 执行命令

提取后按项目/模块分组，识别：
- 新功能开发（feat 开头）
- Bug 修复（fix 开头）
- 重构优化（refactor/perf 开头）
- 文档更新（docs 开头）
- 其他工作

**【确认节点】** 提取完成后，展示给用户确认：
```
从 git 提取到以下工作内容：
1. feat: 完成用户登录模块
2. fix: 修复支付回调异常
3. ...

以上是否完整？如有遗漏请补充。
```

### 第二步：分析与分类

将收集到的工作内容分为两大类：

**核心重点工作**（详细描述）
- 项目关键里程碑
- 重要功能开发
- 线上问题修复
- 技术难题攻克

**日常常规事务**（简要提及）
- 会议、评审、沟通
- 文档整理、代码审查
- 日常运维、答疑

**【确认节点】** 分类完成后，展示分类结果：
```
已将工作内容分类为核心重点工作：
1. 用户中心改版（核心）
2. 修复 3 个线上 bug（核心）
3. 参加需求评审（常规）

分类是否准确？需要调整吗？
```

### 第三步：量化成果

为每项工作补充量化数据。如果用户未提供具体数字：

1. **从 git 提取**：统计 commit 数量、文件变更数、代码行数
2. **模糊量化**：使用"多个"、"若干"、"X 项"等表述
3. **标注待补充**：在周报中用 `[待补充]` 标记，提醒用户后续补充

量化维度：
- 修复了 X 个 bug
- 性能提升 X%
- 代码行数/文件数
- 影响范围（用户数、模块数）
- 完成百分比

### 第四步：生成周报

使用以下标准模板输出：

```markdown
## 本周工作完成

### 核心重点工作

1. 【项目A】完成 XX 模块开发，实现 XX 功能，处理 XX 个需求，已交付测试
2. 【项目B】修复线上 XX 问题，影响 XX 用户，响应时间缩短 XX%
3. ...

### 日常常规事务

1. 参与 X 次评审会议，输出 X 条评审意见
2. 完成 X 个 code review
3. ...

## 现存问题与处理

1. **问题描述**
   - 已尝试：采取的解决措施
   - 需协助：需要的支持和资源

（如无问题可写"本周无重大阻塞问题"）

## 下周工作计划

### 高优

1. 【任务名称】具体交付内容，预计完成时间
2. ...

### 常规

1. 【任务名称】具体工作内容
2. ...

## 其他补充

（可选：学习沉淀、流程优化建议、请假/外出说明等）
```

**模板变体：**
- `--brief` 简版：仅包含"本周完成"+"下周计划"（适合日常快速汇报）
- `--detailed` 详细版：完整四段式（适合周会正式汇报，默认）

## 撰写原则

### 必须遵循

1. **主次分明**：核心工作详细写（3-5 句），常规工作一笔带过（1 句）
2. **全部量化**：用数字支撑，避免"对接"、"优化"、"整理"等空洞词汇
3. **成果导向**：写"完成了什么"，而非"做了什么"
4. **计划落地**：下周计划必须有具体交付物和时间节点

### 必须避免

1. **流水账**：逐日记录琐事，无重点
2. **空洞无物**：全文无量化数据
3. **只报问题**：只说困难不提解决方案
4. **模糊计划**：下周"继续跟进XX"、"处理日常事务"

## 示例转换

### 示例 1：对话描述转周报

**用户输入（零散描述）：**
> 这周主要在做用户中心的改版，写了登录和注册两个页面，还修了几个线上 bug，参加了一次需求评审。

**输出（标准周报）：**

```markdown
## 本周工作完成

### 核心重点工作

1. 【用户中心】完成登录、注册页面改版开发（2 个页面，含表单验证、接口对接），已提交测试
2. 【线上修复】修复 3 个线上 bug（含 1 个 P0 级支付异常），已验证上线

### 日常常规事务

1. 参与用户中心二期需求评审，输出 5 条评审意见

## 现存问题与处理

本周无重大阻塞问题

## 下周工作计划

### 高优

1. 【用户中心】完成个人中心、账户设置页面开发，预计周三交付测试

### 常规

1. 配合测试完成登录注册模块验收
2. 输出用户中心前端开发文档
```

### 示例 2：Git 提交记录转周报

**git log 输出：**
```
a1b2c3d feat(auth): implement JWT token refresh mechanism
d4e5f6g fix(payment): resolve callback timeout issue
h7i8j9k refactor(api): optimize database query performance
l0m1n2o docs(readme): update API documentation
p3q4r5s feat(dashboard): add real-time data visualization
```

**输出（标准周报）：**

```markdown
## 本周工作完成

### 核心重点工作

1. 【认证模块】实现 JWT token 自动刷新机制，提升登录态稳定性，涉及 3 个文件 150+ 行代码
2. 【支付模块】修复回调超时问题，影响约 5% 的支付交易，已验证上线
3. 【仪表盘】新增实时数据可视化功能，支持 5 种图表类型，已提测

### 日常常规事务

1. 重构 API 层数据库查询，优化查询性能
2. 更新 API 文档，补充新增接口说明

## 现存问题与处理

本周无重大阻塞问题

## 下周工作计划

### 高优

1. 【仪表盘】配合测试完成验收，预计周四上线

### 常规

1. 持续优化 API 性能，目标减少 20% 响应时间
```

## 周报质量检查

生成周报后，自动检查以下质量指标：

| 检查项 | 合格标准 | 不合格处理 |
|---|---|---|
| 核心工作数量 | ≥ 1 项 | 提示用户补充重点工作 |
| 量化数据覆盖 | ≥ 50% 的工作项有数字 | 标注 `[待补充]` 提醒用户 |
| 空洞词汇 | 无"对接"、"整理"、"优化"等 | 替换为具体成果描述 |
| 下周计划 | 高优任务 ≥ 1 项 | 提示用户补充高优任务 |
| 计划具体性 | 有交付物和时间节点 | 提示用户明确"做什么"+"何时完成" |

如检查不通过，展示问题清单并询问用户是否需要调整。

## 注意事项

- 如果用户提供了多个来源（对话 + git + 文件），合并去重后整合
- 周报中的"问题与处理"部分鼓励用户填写，但不强制
- 下周计划引导用户按优先级分类，但如用户未提供，可标注"[待补充]"
- 保持语言简洁专业，避免口语化表达
- 时间范围默认为本周一至今天，用户可指定其他范围
