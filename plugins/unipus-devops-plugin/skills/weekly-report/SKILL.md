# 工作周报转写技能

帮用户想清楚哪些事值得写、怎么写出价值感——而不只是把工作内容重新排版。

**核心能力：** 自动判断工作优先级，把"做了什么"翻译成"带来了什么"。

---

## 第一步：收集工作原材料

从以下来源获取本周工作信息（可组合使用）：

**来源 1：对话描述**
- 如果用户已描述工作内容，从对话历史中提取
- 如果用户只说"帮我写周报"而未提供内容，询问：
  > "请简要描述本周的主要工作内容，包括完成的项目、修复的问题、参与的会议等"

**来源 2：工作日志文件**
如果用户提供了日志文件，读取并提取关键工作项。

**来源 3：Git 提交记录（推荐）**

```bash
# 由 Claude 计算本周一日期，替换 YYYY-MM-DD
git log --since="YYYY-MM-DD" --until="now" --oneline --author="$(git config user.name)"

# 获取含文件变更的详细信息
git log --since="YYYY-MM-DD" --until="now" --stat --author="$(git config user.name)"
```

边界处理：
- 非 git 仓库：提示切换目录或跳过
- 本周无提交：提示通过对话补充
- 指定仓库路径：使用 `git -C <path>`

---

## 第二步：自动优先级判断 + 价值翻译

这是核心步骤。拿到原材料后，Claude 独立完成以下判断，**不需要用户回答问题**：

### 优先级判断

按以下信号自动排序，高优先级的放前面：

| 信号 | 权重 |
|---|---|
| P0/P1 线上问题修复 | 最高 |
| 直接影响用户/收入/业务指标 | 高 |
| 关键功能交付（里程碑） | 高 |
| 技术风险消除（安全、性能瓶颈） | 中 |
| 常规功能开发 | 中 |
| 重构、文档、code review、会议 | 低 |

低优先级的多项工作合并为一条，不单独占行。

### 价值翻译

把每条工作从**行为语言**翻译成**价值语言**。翻译框架：

```
行为语言：做了什么（过程）
价值语言：带来了什么（结果 + 受益方）
```

**翻译规则：**

- 有数据就用数据：提交数、影响用户数、性能指标、完成百分比
- 没有数据用状态词：`已上线` `已提测` `已交付` `已验收`
- **绝不猜测数字**：git log 里没有"影响 5% 用户"这类信息，不要编造
- 用 `→` 连接行为和价值

**翻译示例：**

```
❌ 完成了用户中心登录页改版
✅ 【用户中心】登录页改版 → 已提测，覆盖表单验证和接口对接

❌ 修复了若干线上 bug
✅ 【支付】修复回调超时（P0）→ 已上线，恢复受影响交易

❌ 参加了需求评审
✅ （合并）参与需求评审、code review 等日常事务

❌ 优化了数据库查询性能
✅ 【API 层】重构数据库查询 → 响应时间从 800ms 降至 120ms（如有数据）
   【API 层】重构数据库查询 → 已上线，查询性能显著提升（无数据时）
```

### 问题 → 方案转化

如果原材料中有问题或阻塞，不写"遇到了 XX 问题"，而是写：

```
【风险】现象 + 我在做什么 + 预计什么时候解决
```

没有方案的问题不写进周报。如果用户提到了困难但没有方案，可在生成后提醒用户补充。

### 下周计划落地化

把模糊计划转化为可验收的交付承诺：

```
❌ 继续跟进用户中心开发
✅ 【用户中心】完成个人中心、账户设置页面开发 · 预计周三提测

❌ 处理日常事务
✅ （不写，无价值）
```

---

## 第三步：单次确认

完成优先级判断和价值翻译后，**只问一个问题**：

> "我整理后，本周最核心的工作是「XX」。这个判断准确吗？还有什么高价值的工作我可能遗漏了？"

用户确认或补充后，直接生成周报，不再追问。

---

## 第四步：生成周报

```markdown
## 本周完成

1. 【模块名】做了什么 → 价值/结果
2. 【模块名】做了什么 → 价值/结果
3. ...
（按价值高低排序；日常事务合并为最后一条）

## 下周计划

1. 【任务名】交付什么 · 预计 [日期]
2. 【任务名】交付什么
（有明确 deadline 的标注日期；无交付物的计划不写）
```

**生成后**附一句："如有遗漏或需要调整，告诉我即可。"

**变体：** 用户说 `--brief` 时去掉注释行，只输出条目（适合直接粘贴到 IM）。

---

## 质量检查

生成后自动核查，不通过时告知用户：

| 检查项 | 合格标准 |
|---|---|
| 每条是否有价值句 | `→` 后有结果，或以"已上线/已提测/已交付"结尾 |
| 是否有数字编造 | 只写有据可查的数据，无数据用状态词代替 |
| 下周计划是否可验收 | 有具体交付物，不含"继续跟进"等模糊表述 |
| 问题是否附带方案 | 有问题必有处理方向，否则不写 |

---

## 多来源合并

用户提供多个来源（对话 + git + 文件）时，合并去重后整合：
- git 提交作为事实基础
- 对话描述补充 git 无法体现的价值（用户影响、业务背景）
- 文件内容补充具体数字或背景

时间范围默认本周一至今天，用户可指定其他范围。

## 示例

### 示例 1：对话描述 → 周报

**用户输入：**
> 这周主要在做用户中心的改版，写了登录和注册两个页面，还修了几个线上 bug，参加了一次需求评审。

**Claude 内部判断：**
- P0 级别：线上 bug 修复（最高优）
- 核心交付：登录注册改版（高优）
- 日常事务：需求评审（低优，合并）

**输出：**

```markdown
## 本周完成

1. 【线上修复】修复 3 个线上 bug（含 1 个 P0 支付异常）→ 已验证上线
2. 【用户中心】完成登录、注册页面改版（含表单验证、接口对接）→ 已提测
3. 参与用户中心二期需求评审

## 下周计划

1. 【用户中心】完成个人中心、账户设置页面开发 · 预计周三提测
2. 配合测试完成登录注册模块验收
```

### 示例 2：git log → 周报

**git log 输出：**
```
a1b2c3d feat(auth): implement JWT token refresh mechanism
d4e5f6g fix(payment): resolve callback timeout issue
h7i8j9k refactor(api): optimize database query performance
l0m1n2o docs(readme): update API documentation
p3q4r5s feat(dashboard): add real-time data visualization
```

**Claude 内部判断：**
- P0 级别：payment fix（最高优）
- 核心功能：auth、dashboard（高优）
- 日常工作：refactor、docs（低优，合并）

**输出：**

```markdown
## 本周完成

1. 【支付】修复回调超时问题（P0）→ 已上线验证
2. 【认证】实现 JWT 自动刷新机制 → 消除登录态意外失效，已提测
3. 【仪表盘】新增实时数据可视化功能 → 已提测
4. 重构 API 查询层，更新接口文档

## 下周计划

1. 【仪表盘】完成测试验收并上线 · 预计周四
2. 持续优化 API 响应性能，目标减少 20% 响应时间
```
