# 文档命名规范

## 文件命名格式

```
<版本>_<项目名称>_<文档类型>.<扩展名>
```

## 版本号规范

| 格式 | 适用场景 |
|---|---|
| `V1.0` | 正式首版 |
| `V2.0` | 大版本更新（架构调整、重大功能变更） |
| `V1.1`、`V1.2` | 小版本迭代（功能补充、细节修改） |
| `V0.1`、`V0.9` | 草稿、评审版本 |
| `Draft` | 初稿，待评审 |

## 后端开发常用文档类型

| 文档类型 | 说明 |
|---|---|
| 架构设计文档 | 系统架构、模块设计、数据库设计、部署方案 |
| 后端API接口文档 | API 接口规范、请求/响应定义、错误码 |
| 技术方案 | 技术选型、方案对比、实施计划 |
| 数据库设计文档 | 表结构、ER 图、索引策略、迁移方案 |
| 部署文档 | 部署流程、环境配置、运维手册 |

## 完整示例

| 场景 | 文件名 |
|---|---|
| 统一认证平台，架构设计文档 V1.0 | `V1.0_统一认证平台_架构设计文档.md` |
| 统一认证平台，后端API接口文档 V1.0 | `V1.0_统一认证平台_后端API接口文档.md` |
| 统一认证平台，新增功能后更新接口文档 V1.1 | `V1.1_统一认证平台_后端API接口文档.md` |
| 翻译平台，技术方案草稿 | `Draft_翻译平台_技术方案.md` |
| K12题库系统，数据库设计文档 V2.0 | `V2.0_K12题库系统_数据库设计文档.md` |

## 新增需求的版本处理

当为已有项目新增功能时：
- **不创建新文件**，在已有的接口文档中追加新接口
- 版本号递增小版本（如 V1.0 → V1.1）
- 在文档顶部的「版本变更记录」中添加变更说明
- 文件名同步更新版本号

## 注意事项

- 项目名称保持一致（不同版本文档使用相同项目名称）
- 避免使用空格，可用下划线 `_` 分隔
- 中文命名优先，英文缩写需全员统一认知后使用
- 扩展名按实际文件格式填写（`.md`、`.docx`、`.pdf` 等）
- 与 `optimus-feishu-project-docs-push` 技能的命名规范保持一致
