# optimus-test-design — 测试设计一站式工具

## 概述

统一入口，支持从 PRD 到测试用例的完整测试设计流水线，也可单独生成任意一环文档。

| 模式 | 触发条件 | 输入 | 输出 |
|------|---------|------|------|
| **0：全流程** | 同时提供需求文档 + 设计稿链接 | PRD 文档 + Figma/MasterGo 链接 | 依次输出 A→B→C→D 全部文档 |
| **A：需求拆分** | "需求拆分"、"提取功能需求"、"分析PRD" | PRD 文档（.docx/.md/.pdf） | `docs/{产品名称}-功能需求拆分.md` |
| **B：测试计划** | "生成测试计划"、"测试计划" | 功能需求拆分.md（必需）+ 架构设计文档（可选） | `docs/{产品名称}_测试计划.md` |
| **C：测试点** | "提取测试点"、"生成测试点"、"测试点分析" | 需求拆分文档 | `docs/testcases/{需求名称}-{模块名称}测试点.md` |
| **D：测试用例** | "生成测试用例"、"测试用例"、"用例设计" | 测试点文档（或需求文档） | `docs/testcases/{需求名称}-测试用例-{序号}.md` |
| **E：合并测试用例** | "合并测试用例"、"汇总用例"、"合并用例"、"生成完整用例" | `docs/testcases/{需求名称}-测试用例-*.md` | `docs/testcases/{需求名称}-测试用例.md` |

完整流水线：PRD → [A]需求拆分 → [B]测试计划 → [C]测试点 → [D]测试用例 → [E]合并测试用例

## Step 0：识别模式

**优先判断全流程触发：**
- 用户同时提供了需求文档（PRD）**且**提供了设计稿链接（Figma/MasterGo）→ **模式 0（全流程）**

**否则按触发词判断单独模式：**
- 含"合并测试用例"/"合并用例"/"汇总用例"/"生成完整用例" → **模式 E**（优先于 D 判断）
- 含"需求拆分"/"提取功能需求"/"分析PRD"/"PRD" → **模式 A**
- 含"测试计划"/"计划" → **模式 B**
- 含"测试点"/"测试要点"/"提取" → **模式 C**
- 含"测试用例"/"用例"/"用例设计" → **模式 D**

若无法判断，询问用户：「请问您要：(A) 提取功能需求、(B) 生成测试计划、(C) 提取测试点、(D) 生成测试用例，还是 (E) 合并所有模块测试用例？」

## 模式 0：全流程自动流水线

用户同时提供需求文档和设计稿时，**按顺序自动执行 A→B→C→D**，无需用户在每步确认，完成后汇报全部输出文件列表。

### 执行顺序

```
Step 1 → 读取设计稿（Figma/MasterGo），按「设计稿描述规范」生成文字描述，保存至 docs/{产品名称}_设计稿描述.md
Step 2 → 执行模式 A：生成功能需求拆分文档
Step 3 → 执行模式 B：基于需求拆分文档生成测试计划
Step 4 → 执行模式 C：基于需求拆分文档生成各模块测试点
Step 5 → 执行模式 D：结合设计稿描述文档生成测试用例（含端到端用例）
```

### 全流程完成汇报格式

```
✅ 全流程已完成，共生成以下文档：
- docs/{产品名称}_设计稿描述.md
- docs/{产品名称}-功能需求拆分.md
- docs/{产品名称}_测试计划.md
- docs/testcases/{需求名称}-{模块名称}测试点.md（共 N 个）
- docs/testcases/{需求名称}-测试用例-01.md ... 测试用例-{N}-端到端流程.md
```

## 设计稿描述规范

**触发时机：** 模式 0 Step 1、模式 D D-Step 0 读取设计稿时，均按本规范生成文字描述文档。

**输出路径：** `docs/{产品名称}_设计稿描述.md`

### 读取步骤

**Figma 设计稿：**
1. 使用 `get_figma_data` 读取文件顶层结构，获取 Canvas 名称和所有顶层 Frame（页面）列表
2. 逐个读取每个 Frame 的详细节点数据（depth=3），提取布局层级、元素名称、文案、状态标注

**MasterGo 设计稿：**
1. 使用 `mcp__getDsl` 读取文件顶层结构，获取所有页面 Frame 列表
2. 逐个读取每个页面的 DSL 节点数据，提取布局结构、组件名称、文案、状态

### 文档格式规范

参考格式：`docs/LMS口语练习_设计稿描述.md`

```markdown
# {产品名称} - 设计稿文字描述

> 来源：{设计稿链接}
> 生成日期：{YYYY-MM-DD}

---

## 整体结构

设计稿名称：**{名称}**，包含 {N} 个 Canvas/页面，简要说明覆盖的用户流程。

---

## 页面 N：{页面名称}

**{区域名称}**（描述该区域整体特征）：
- {元素1}：{内容/功能描述}
- {元素2}：{内容/功能描述}

**{子区域名称}**（描述特征）：
- {元素}：{描述}

---

## 设计规范摘要

- 主色调：{颜色描述}（仅此处可标注色值）
- 等级/状态颜色体系：{描述}
- 字体：{字体名}，字重范围
- 卡片圆角：{值}，{阴影描述}
- 整体风格：{简要描述}
```

### 描述内容规范

**必须描述的内容：**
- 每个页面/状态的名称（直接从设计稿 Frame 名称提取）
- 顶部导航栏：标题文案、左右操作项、动态数值（如进度 0/10）
- 主内容区：卡片数量、卡片内元素名称和文案
- 底部操作栏：所有按钮名称和当前状态（高亮/禁用/全宽）
- 弹窗/模态层：标题、说明文字、按钮文案
- 数据展示区：标签、数值、图表类型（进度条/圆环/柱状图）

**描述层级规则：**
- 用 `**粗体**` 标注区域名称（导航栏、卡片、弹窗、操作栏）
- 区域下的元素用 `-` 列表逐项描述
- 嵌套子元素用缩进 `-` 列表
- 同一页面多个卡片用 `卡片 N -` 前缀区分

**状态标注规则：**
- 按钮状态必须标注：`（高亮可点击）`、`（禁用状态）`、`（全宽）`
- 动态内容必须标注具体值：进度「1/10」、分数「73 分」、时间「1s / 5s」
- 特殊视觉元素用括号补充：`（红色圆点）`、`（带阴影）`、`（渐变背景）`

**禁止在页面描述中出现的内容（仅设计规范摘要章节允许）：**
- 具体色值（#6366F1、rgb(...)）
- 具体像素值（48px、圆角24px、160×160px）
- CSS 属性名（font-weight、border-radius、box-shadow）

### 质量检查

- [ ] 文档开头包含设计稿来源链接和生成日期
- [ ] 整体结构章节说明了设计稿名称和页面数量
- [ ] 所有 Frame（页面/状态）均有对应章节，无遗漏
- [ ] 每个页面描述了导航栏、主内容区、操作栏三大区域
- [ ] 按钮文案、状态（高亮/禁用）、动态数值均已标注
- [ ] 页面描述中无具体色值、无像素数值、无 CSS 属性名
- [ ] 文末包含「设计规范摘要」章节（主色调、等级颜色、字体、圆角、整体风格）
- [ ] 输出路径：`docs/{产品名称}_设计稿描述.md`

## 模式 A：需求拆分

### 核心概念

将 PRD 中非结构化的产品描述，转化为**原子级、可验证、可追溯**的功能需求条目，供测试人员直接使用。

### A-Step 1：通读 PRD，识别功能模块

- 理解产品定位、目标用户和核心价值
- 识别所有独立功能模块（通常对应 PRD 章节）
- 为每个模块编号：`功能模块1`、`功能模块2`...
- 模块命名简洁明确，反映核心功能

### A-Step 2：逐模块提取四大要素

每个功能模块提取：

| 要素 | 内容要求 |
|------|---------|
| **功能概述** | 1-2句概括核心能力和业务价值，用户视角，不含技术实现 |
| **用户故事** | 格式：`作为[角色]，我希望能够[操作]，[目的]`，每模块3-5条 |
| **功能需求（FR）** | 原子级需求，全局连续编号 FR-001、FR-002…，按子功能分组 |
| **验收标准（AC）** | 全局连续编号 AC-001…，包含可量化指标 |

### A-Step 3：提取非功能需求

所有功能模块完成后，单独提取附录，使用 `NFR-` 编号，分五类：性能指标、质量指标、兼容性要求、安全性要求、可用性要求。

### A-Step 4：生成目录与变更记录

- 文档开头：各功能模块锚点链接目录
- 文档末尾：版本/日期/修改人/修改内容变更表格

### FR 拆解规则

**粒度标准**：一条 FR = 一个可独立测试的功能点

- 包含"和"、"并且"的复合需求 → 拆分为多条
- 多个校验规则 → 每条规则独立一条 FR
- 文件上传类：格式、大小、编码、时长等各自独立

> 详细拆解规则见 [references/decomposition-rules.md](references/decomposition-rules.md)

### 输出格式

```markdown
# [产品名称] - 功能需求拆分文档
> 编号范围：FR-001 ~ FR-XXX | AC-001 ~ AC-XXX | NFR-001 ~ NFR-XXX

## 目录
- [功能模块1: XXX](#功能模块1-xxx)

---

## 功能模块N: [模块名称]

### 1. 功能概述
### 2. 用户故事
### 3. 功能需求
#### 3.1 [子功能名称]
- FR-XXX: [功能需求描述]
### 4. 验收标准
- AC-XXX: [含可量化指标的验收标准]

---

## 附录：非功能需求
- NFR-XXX: [要求描述]

## 文档变更记录
```

> 完整模板见 [references/prd-output-template.md](references/prd-output-template.md)

### A-质量检查

- [ ] PRD 所有功能点都已提取为 FR
- [ ] 每个模块包含概述、用户故事、功能需求、验收标准四部分
- [ ] FR/AC/NFR 编号全局连续无遗漏
- [ ] 每条 FR 只描述一个功能点，含"和/并且"的复合需求已拆分
- [ ] 验收标准包含可量化指标
- [ ] 输出路径：`docs/{产品名称}-功能需求拆分.md`

## 模式 B：测试计划

### 输入文档要求

| 输入文档 | 必要性 | 用途 |
|---------|--------|------|
| 功能需求拆分文档 | 必需 | 提取测试范围、功能模块、FR/AC/NFR条目 |
| 系统架构设计文档 | 推荐 | 提取技术架构、接口清单、第三方依赖、业务流程 |

### B-Step 1：解析输入文档，提取关键信息

从功能需求拆分文档中提取：
- 所有功能模块名称和编号
- 各模块的FR条目数量和关键功能点
- AC验收标准中的量化指标
- NFR非功能需求

从架构设计文档中提取（如有）：
- 技术架构分层、接口清单和接口数量
- 第三方服务依赖列表、核心业务流程、安全设计要点

### B-Step 2：生成测试计划各章节

按以下 11 章节顺序生成，详细规则见 [references/testplan-chapter-rules.md](references/testplan-chapter-rules.md)，输出模板见 [references/testplan-output-template.md](references/testplan-output-template.md)。

**输出文档结构：**
```
1. 项目概述（背景、功能概述、测试目标）
2. 测试范围（In Scope / Out of Scope）
3. 测试策略（测试类型、测试方法、测试优先级）
4. 测试环境（硬件、软件、测试数据、网络）
5. 测试用例设计（各模块用例概要、设计原则、数据设计）
6. 测试执行计划（阶段划分、详细时间安排甘特图）
7. 测试资源（人员配置、技能要求、工具和设备）
8. 风险评估（技术风险、项目风险、应对措施）
9. 准入/准出标准
10. 测试交付物
11. 测试注意事项
```

**关键生成规则摘要：**

- **测试类型**：有功能模块→功能测试；NFR含性能→性能测试；NFR含安全→安全测试；架构含第三方→集成测试；始终含回归测试
- **工作量估算**：FR条目数÷5=功能测试人天；接口数÷8=接口测试人天；性能测试固定2-3天；安全测试固定2-3天
- **排期策略**：小型（FR≤50）约8天；中型（FR 50-150）约15-22天；大型（FR>150）约34-46天

### B-质量检查

- [ ] 所有功能模块都纳入测试范围
- [ ] NFR中的所有指标都有对应的测试类型覆盖
- [ ] 测试计划包含全部11个章节
- [ ] 工作量估算合理，人员配置与工作量匹配
- [ ] 输出路径：`docs/{产品名称}_测试计划.md`

## 模式 C：测试点

### C-Step 1：需求分析与 ID 规范化

逐条读取需求，若无ID，按以下格式生成：
```
REQ_[模块代码]_[类型代码]_[序号]
```

类型代码：`FUNC`(功能) / `PERF`(性能) / `SEC`(安全) / `UI`(界面) / `DATA`(数据) / `INTG`(集成)

### C-Step 2：功能点划分

按业务逻辑将需求拆分为独立功能点，遵循层级结构：
```
模块 → 功能点 → 验证类型 → 具体测试点
```

### C-Step 3：测试点提取

在每个功能点下，按四类验证类型生成测试点：

| 验证类型 | 关注点 | 典型场景 |
|----------|--------|----------|
| **功能验证点** | 核心业务逻辑 | CRUD操作、业务动作、状态变更 |
| **边界验证点** | 边界条件 | 输入长度上下限、权限边界、数据量限制 |
| **异常验证点** | 异常处理 | 错误输入、网络异常、权限不足 |
| **集成验证点** | 模块协作 | 跨模块数据传递、外部接口调用 |

详细提取规则见 [references/testpoint-extraction-rules.md](references/testpoint-extraction-rules.md)

### C-Step 4：生成文档

**输出格式：**
```markdown
- TP_[模块]_[序号]: [测试点名称]
  - 验证要点: [具体验证内容]
  - 优先级: 高/中/低
  - 关联需求: REQ_XXX_XXX_XXX
  - 派生测试用例: [预计数量]个
```

文档末尾必须包含：测试点统计、需求追溯矩阵、需求覆盖率统计。

完整模板见 [references/testpoint-templates.md](references/testpoint-templates.md)

### C-质量检查

- [ ] 所有需求都有对应测试点（无遗漏）
- [ ] 每个功能点涵盖正常、边界、异常场景
- [ ] 追溯矩阵无遗漏，覆盖率已统计
- [ ] 输出路径：`docs/testcases/{需求名称}-{功能模块名称}测试点.md`

## 模式 D：测试用例

### D-Step 0：确认设计稿

**若在全流程模式（模式0）中已生成 `{产品名称}_设计稿描述.md`，直接读取该文件，跳过此步。**

**单独触发时必须询问用户：**

> "是否有 Figma/MasterGo 设计稿链接？如果有，请提供链接，我会按「设计稿描述规范」读取并生成文字描述文档，再结合描述生成更准确的测试用例。"

**如果用户提供了设计稿链接：**
1. 按「设计稿描述规范」完整执行读取和描述流程
2. 输出 `docs/{产品名称}_设计稿描述.md`
3. 后续生成测试用例时，以设计稿描述文档中的页面名称、元素名称、按钮文案作为测试步骤和预期结果的描述依据

**如果用户没有设计稿：** 直接进入 D-Step 1。

### D-Step 1：解读测试点结构

测试点文档层级：`功能模块 → 功能点 → 验证类型 → 具体测试点（含验证要点）`

### D-Step 2：按验证类型映射用例策略

| 测试点类型 | 用例设计策略 |
|------------|-------------|
| 功能验证点 | 正向用例——验证功能正常工作 |
| 边界验证点 | 边界用例——设计多组边界值数据（最小值、最大值、临界值、超界值） |
| 异常验证点 | 异常用例——覆盖各种错误处理场景 |
| 集成验证点 | 组合用例——验证跨模块交互 |
| 端到端验证 | 流程用例——覆盖完整业务路径的跨模块串联场景（见 D-Step 5） |

### D-Step 3：细化为具体场景

- 一个测试点对应**一到多个**用例场景
- 将验证要点中的抽象描述转换为具体操作步骤和测试数据
- **每个功能模块生成约 20 条关键用例**，聚焦核心场景，避免冗余

> **⚠️ 全局禁止规则：测试用例全文（含前置条件、测试步骤、预期结果）禁止包含任何 UI 样式描述**，包括但不限于：颜色值（#xxx、rgb）、字号（14px、16px）、字重（bold）、间距（margin、padding）、圆角（border-radius）、阴影、CSS 属性名。只描述功能行为和内容，不描述视觉实现细节。

### D-Step 4：输出文档

按严格的六级 Markdown 层级输出，兼容 XMind 导入：

```
H1 # 文档标题
H2 ## 功能模块
H3 ### 功能测试点
H4 #### 验证点
H5 ##### [需求ID] 用例场景描述[优先级]
H6 ###### 前置条件（有前置依赖时必须写）
H6 ###### 测试步骤
H6 ###### 预期结果
```

详细格式规范见 [references/testcase-format-spec.md](references/testcase-format-spec.md)，转换示例见 [references/testcase-examples.md](references/testcase-examples.md)

**H5 标题格式：**
```
##### [需求ID] 用例场景描述[优先级]
```
- 需求ID放最前面，方括号包裹，多个用逗号分隔：`[FR-064,AC-032]`
- 优先级放最后面：`[P0]`/`[P1]`/`[P2]`

**优先级分级：**
- P0：核心业务流程、安全认证、数据完整性、权限控制
- P1：重要功能验证、边界条件、关键异常处理、性能指标
- P2：UI细节、低优先级异常、动画效果、空状态展示

### D-Step 5：生成端到端测试用例

在完成各功能模块的测试用例后，**额外生成一份端到端（E2E）流程测试用例文档**，覆盖跨模块的完整业务流程。

**输出路径：** `docs/testcases/{需求名称}-测试用例-{最后序号+1}-端到端流程.md`

**端到端用例设计策略：**

| 场景类型 | 优先级 |
|----------|--------|
| 正常完整流程（所有模块串联主路径） | P0 |
| 混合操作流程（穿插跳过、重试等分支） | P0 |
| 中断恢复流程（中途退出后恢复完成） | P0 |
| 多轮累计流程（连续多次，验证数据累计） | P0 |
| 权限角色流程（不同角色完整访问路径） | P0 |
| 网络异常恢复流程 | P1 |
| 设备异常恢复流程 | P1 |
| 与现有系统集成流程 | P1 |

- 每条用例覆盖**至少3个功能模块**的串联交互
- 约生成 **8-12 条**端到端用例
- 需求ID关联该流程涉及的多个模块核心需求

### D-质量检查

- [ ] 确认是否有设计稿：全流程模式已自动生成描述文档；单独模式已询问用户并按「设计稿描述规范」生成 `{产品名称}_设计稿描述.md`
- [ ] 每个测试点都有对应的测试用例（无遗漏）
- [ ] 每个功能模块约 20 条关键用例
- [ ] H5 标题包含需求ID（前置）和优先级（后置）
- [ ] 整个用例均不包含任何 UI 样式描述（颜色、圆角、字号、间距等）
- [ ] 已生成端到端流程测试用例文档
- [ ] 端到端用例每条覆盖至少3个功能模块的串联交互
- [ ] 输出路径：`docs/testcases/{需求名称}-测试用例-{序号}.md`

## 模式 E：合并测试用例

将 `docs/testcases/` 下的所有分模块测试用例文件合并为一份完整文档，便于统一查阅、导出或上传。

### E-Step 1：识别待合并文件

1. 扫描 `docs/testcases/` 目录，匹配文件名规则：`{需求名称}-测试用例-{序号}*.md`
   - 排除文件名不含序号的文件（如 `{需求名称}-测试用例.md`，即合并结果文件本身）
   - 排除非 `.md` 文件（如 `.xmind`）
2. 按序号数字升序排列（01、02、03 … 08）
3. 若用户指定了需求名称，只合并匹配该名称前缀的文件；否则自动从文件名推断

### E-Step 2：读取并合并内容

1. 逐个读取排序后的文件
2. 提取每个文件的正文内容（去掉 H1 标题行，保留 H2 及以下所有内容）
3. 各模块内容之间插入分隔线 `---`
4. 统计合并后的总用例数（H5 标题数量）

### E-Step 3：生成合并文档

**输出文件头部格式：**
```markdown
# {需求名称} - 完整测试用例

> 合并自 {N} 个模块文件，共 {M} 条测试用例
> 生成日期：{YYYY-MM-DD}
> 文件列表：
> - {文件1}
> - {文件2}
> - ...

---
```

**合并规则：**
- 每个模块的 H2 标题保留，作为合并文档中的模块分隔标识
- 不修改任何用例内容，只做拼接
- 文件间插入 `---` 分隔线

**输出路径：** `docs/testcases/{需求名称}-测试用例.md`

> ⚠️ 若同名文件已存在，直接覆盖（合并结果文件每次重新生成）

### E-质量检查

- [ ] 所有分模块文件均已读取，无遗漏
- [ ] 文件按序号升序排列，顺序正确
- [ ] 合并文档头部包含文件列表和用例总数
- [ ] 各模块之间有 `---` 分隔线
- [ ] 输出路径：`docs/testcases/{需求名称}-测试用例.md`
