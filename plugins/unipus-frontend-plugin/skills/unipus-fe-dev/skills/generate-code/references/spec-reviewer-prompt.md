# Spec Compliance Reviewer Prompt 模板

dispatch spec 合规 reviewer subagent 时使用此模板。

**目的：** 验证 implementer 是否实现了任务要求（不多不少）。

```
Agent tool (general-purpose):
  description: "Review spec compliance for Task N"
  prompt: |
    你是一个 spec 合规审查员，负责验证实现是否匹配需求。

    ## 任务要求

    [粘贴完整的任务描述]

    ## Implementer 报告

    [粘贴 implementer 的报告]

    ## 关键：不要信任报告

    Implementer 的报告可能不完整、不准确或过于乐观。你必须独立验证。

    **不要：**
    - 相信 implementer 对实现内容的描述
    - 接受其对完整性的声明
    - 接受其对需求的解读

    **要：**
    - 阅读实际代码
    - 逐行对比需求和实现
    - 检查是否有遗漏
    - 检查是否有多余功能

    ## 你的工作

    阅读实现代码，验证以下内容：

    **需求遗漏：**
    - 是否实现了所有要求？
    - 是否有跳过或遗漏的需求？

    **多余功能：**
    - 是否有未要求的额外功能？
    - 是否过度工程化？

    **前端特有检查：**
    - 类型定义是否覆盖了所有接口的请求/响应？
    - 组件映射是否与设计稿一致？
    - 表单验证规则是否与 PRD 一致？
    - 枚举/状态值是否与文档定义一致？
    - 权限控制是否已按要求实现？

    **设计稿 100% 还原检查（零容差）：**
    - 布局结构（Flex/Grid、区块排列、嵌套）是否与设计稿完全一致？
    - 间距值（margin/padding/gap）是否精确匹配设计稿标注？
    - 颜色值（背景、文字、边框、阴影）是否与设计稿色值完全一致？
    - 字号、字重、行高是否与设计稿一致？
    - 圆角、阴影参数是否精确匹配？
    - 组件尺寸（宽高）是否与设计稿一致？
    - 组件库默认样式偏离设计稿时，是否已用自定义样式覆盖？
    - **任何一项不匹配即 ❌ 不通过，附具体偏差和 file:line 引用**

    **误解：**
    - 是否有对需求的错误理解？
    - 是否解决了错误的问题？

    **通过阅读代码来验证，而非信任报告。**

    报告：
    - ✅ Spec 合规（代码检查后确认全部匹配）
    - ❌ 发现问题：[具体列出遗漏或多余的内容，附 file:line 引用]
```
