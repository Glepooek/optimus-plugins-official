使用 `prd-creator` 从零创建标准 PRD 文档。

---

<HARD-GATE>
**开始任何创建工作前，必须先执行以下两步：**
1. Read `references/prd-template.md` — 获取7章结构和每章子节格式
2. Read `references/prd-checklist.md` — 获取6维度质量标准

未读取以上文件不得开始生成。
</HARD-GATE>

## Workflow

**Step 1 — 宣告并读取参考文件**
输入：用户发起的PRD创建请求；输出：已读取的 `prd-template.md`（7章结构）与 `prd-checklist.md`（6维度标准）
告知用户："正在使用 prd-creator"，然后读取 `references/prd-template.md` 和 `references/prd-checklist.md`。

**Step 2 — 收集核心输入**
输入：用户的原始请求（是否已包含功能/需求描述）；输出：用户提供的功能/需求描述原始文本
如果用户已提供功能/需求描述，直接进入 Step 3。
否则，问用户第一个问题：
> "请描述您要创建 PRD 的功能或需求是什么？"

**Step 3 — 逐一追问缺失关键信息**
输入：Step 2 得到的功能/需求描述；输出：补全后的六类关键信息，或（用户拒绝追问时）经CHECKPOINT分类处理后的候选方案+待确认标注
根据已有信息，判断以下内容是否缺失，每次只问 1 个问题，最多追问 6 次：
- 需求背景（为什么要做？现状是什么？）
- 目标用户（谁在用？）
- 核心功能点（做什么？）
- 需求目标与衡量指标（成功标准是什么？）
- 非功能需求（性能/安全/兼容性要求？）
- 上线时间或优先级约束

若某项信息可从上下文合理推断，则跳过该问题。

**若用户明确拒绝追问**（如说"别问了""不用问""直接生成一版"）：

🔴 **CHECKPOINT — 生成前必须先完成信息分类，不得直接生成：**
1. 把仍缺失的信息按来源分类，判定规则是封闭且确定的（不需要主观判断"是否影响功能正确性"）：
   - **业务决策类** = Step 3 的全部六类信息（需求背景、目标用户、核心功能点、需求目标与衡量指标、非功能需求、上线时间或优先级约束）——这六类本身就是 Step 3 专门设计用来收集的实质性需求内容，无论具体缺哪一项，都归业务决策类
   - **行政元数据类** = 与 Step 3 无关、来自第1章"文档信息"模板的字段（如负责人、关联文档链接、版本历史、文档编号）
2. 业务决策类缺失 → 基于上下文给出一个具体候选方案，标注"⚠️ 待确认：[候选方案] + [推断依据]"，不得留空
3. 行政元数据类缺失 → 直接用简单占位符填充，不与业务决策类混标、不单独强调
4. Step 6 自检报告开头必须统计"业务决策类待确认项"数量；超过 3 项时须在报告最前面提示："本次因未追问信息，以下 N 项关键业务决策为 AI 假设，强烈建议人工复核后再用于开发"

**六类信息收集/推断完毕后，进入 Step 4 前必须先做一次矛盾检查（用户提供的原始信息、AI自行推断的候选方案，一视同仁纳入检查范围）：**

🔴 **CHECKPOINT — 若发现信息前后矛盾，禁止假装没发现直接生成：**
- 检查方法：不要只扫描字面对立词（如"实时"vs"离线"这类表面关键词冲突），要审视每对信息背后隐含的技术/业务假设是否互相兼容；对本轮由AI自己推断生成的候选方案要用比检查用户原话更高的怀疑度审视——自己刚编的答案容易被自己的措辞说服，即使几项候选方案字面用词方向一致，也不代表背后假设不冲突
- 检查范围：需求背景陈述的现状 vs 需求目标与衡量指标的方向是否一致；核心功能点 vs 非功能需求是否存在冲突（例："多人实时编辑+冲突自动合并"隐含的是最终一致性模型，与"强一致性/不产生覆盖性冲突"这类表述存在真实工程矛盾——即使两者字面都在强调"实时""不丢数据"，听起来方向一致）；目标用户 vs 核心功能点的适用范围是否匹配
- 若发现矛盾 → 停下，用一句话向用户说明具体矛盾点（引用双方原话或候选方案原文），请求澄清，不得自行选择一方推进
- 若未发现矛盾 → 直接进入 Step 4

**Step 4 — 使用 TodoWrite 跟踪生成进度**
输入：Step 3 收集/推断出的六类关键信息；输出：TodoWrite 任务清单（8项，覆盖7章生成+内置自检）
创建任务清单：
- [ ] 生成第1章：文档信息
- [ ] 生成第2章：需求概述
- [ ] 生成第3章：用户分析
- [ ] 生成第4章：功能需求
- [ ] 生成第5章：非功能需求
- [ ] 生成第6章：验收标准
- [ ] 生成第7章：其他
- [ ] 内置质量自检

**Step 5 — 生成 PRD 草稿**
输入：Step 3/CHECKPOINT 阶段得到的完整信息 + `prd-template.md` 的7章结构；输出：全7章PRD草稿，子节均有实质内容，无占位符
严格对照 `references/prd-template.md` 的结构生成完整 PRD，包含全部7章及子节。所有章节必须有实质内容，不得保留空白占位符。

**Step 6 — 内置质量自检（相当于运行 prd-reviewer）**
输入：Step 5 生成的PRD草稿 + `prd-checklist.md` 的6维度标准；输出：质量自检报告（6维度打分+综合得分），综合得分<70时输出修订后的新草稿
对照 `references/prd-checklist.md` 的6维度逐项评分，输出质量报告：

```
质量自检报告
├── 结构完整性：XX/100
├── 内容详细度：XX/100
├── 逻辑一致性：XX/100
├── 可执行性：XX/100
├── 规范性：XX/100
├── AI友好性：XX/100
└── 综合得分：XX/100

待改进项：
- [问题描述] → [改进建议]
```

若综合得分低于 70，自动修订 PRD 后重新输出。

**Step 7 — 保存文档**
输入：Step 6 通过自检（或已修订）的最终PRD内容；输出：保存到 `docs/prd/{功能名}.md` 的文件，及给用户的保存路径说明
将完整 PRD 保存到 `docs/prd/{功能名}.md`，文件名使用功能名称的英文小写或拼音缩写（如 `user-login.md`）。

🔴 **CHECKPOINT — 写入前必须先检查目标路径是否已存在同名文件，禁止静默覆盖：**
- 若 `docs/prd/{功能名}.md` 不存在 → 直接写入
- 若已存在 → 停下告知用户："检测到 `{路径}` 已存在，是否覆盖？(1) 覆盖 (2) 另存为新文件名 (3) 取消保存，仅在对话中展示PRD内容"，等用户选择后才执行对应操作，不得在未确认前写入
- 选择"另存为新文件名"时，命名规则为原文件名追加 `-v2`（如 `-v2` 已存在则递增为 `-v3`，以此类推），不得另行询问用户具体命名

## Red Flags

| 错误做法 | 正确做法 |
|---------|---------|
| 一次性问多个问题 | 每次只问1个问题，最多6个 |
| 保留模板占位符（如"[说明]"）输出 | 所有章节填入实质内容 |
| 跳过非功能需求章节 | 5个子节均需覆盖，无数据时注明"待确认" |
| 验收标准与需求目标不对应 | 第6章每条验收标准必须能追溯到第2.2节的目标 |
| 跳过内置自检 | Step 6 是强制步骤，不得跳过 |
| 追问超过6个问题 | 超过6个问题后基于现有信息推断并注明假设 |
| 用户拒绝追问时，把行政元数据和业务决策不分主次全标"待确认" | 只对业务决策类标"⚠️ 待确认"+给候选方案，行政元数据类直接留空不强调 |
| 保存前不检查同名文件是否已存在，直接覆盖 | Step 7 写入前必须先检查，已存在时停下询问用户覆盖/另存/取消 |
| 收集到的信息前后矛盾（如背景与衡量指标方向相反）仍照常生成 | Step 3 末尾必须先做矛盾检查，发现矛盾停下澄清，不自行选边 |
