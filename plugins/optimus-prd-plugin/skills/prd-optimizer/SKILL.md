使用 `prd-optimizer` 对已有 PRD 进行结构增强和内容补全。

---

<HARD-GATE>
**开始任何优化工作前，必须先执行以下两步：**
1. Read `references/prd-template.md` — 获取标准7章结构，作为补全依据
2. Read `references/prd-checklist.md` — 获取6维度评估标准，作为优化方向

未读取以上文件不得开始优化。
</HARD-GATE>

---

## Workflow

**Step 1 — 宣告并读取参考文件**
告知用户："正在使用 prd-optimizer"，然后读取 `references/prd-template.md` 和 `references/prd-checklist.md`。

**Step 2 — 获取待优化 PRD**
- 用户已提供文件路径 → 用 Read 工具读取文件内容
- 用户直接粘贴内容 → 直接使用粘贴内容
- 未提供任何输入 → 问用户：
  > "请提供需要优化的 PRD 文档路径或直接粘贴文档内容。"

**Step 3 — 内置 Review（问题诊断）**
对照 `references/prd-checklist.md` 快速评估，输出问题摘要（不需要完整报告格式，只需列出关键问题）：

```
当前文档问题摘要
├── 综合得分（估算）：XX/100
├── 结构缺失：[列出缺失章节/子节]
├── 内容不足：[列出内容薄弱项]
├── 逻辑问题：[列出矛盾或不一致项]
└── 可执行性缺口：[列出开发/测试无法依赖的部分]
```

**Step 4 — 使用 TodoWrite 跟踪优化进度**
创建任务清单，包含需要优化的具体章节。

**Step 5 — 询问优化重点**

🔴 **CHECKPOINT**——用户在本次请求中若已明确指定优化重点，可跳过提问直接进入 Step 6；否则必须停下询问，禁止自行假设后继续执行：

问用户（单选或多选）：
> "请选择优化重点（可多选）：
> A. 结构完整性（补全缺失章节）
> B. 内容详细度（丰富功能描述、业务规则）
> C. 可执行性（增强开发/测试可依据性）
> D. 全面优化（A+B+C）"

**Step 6 — 执行优化**
根据用户选择，对照 `references/prd-template.md` 执行对应优化：
- 结构优化：按模板补全缺失章节和子节
- 内容优化：丰富业务规则、交互细节、异常处理、埋点等
- 可执行性优化：补充验收标准、边界条件、具体数值
- 全面优化：以上全部执行

**原则**：保持原有功能意图不变，只做增强补全，不删除原有内容；**仅执行用户选中的选项对应的优化动作**——若诊断时发现属于其他未选中选项的缺口（如用户只选C可执行性，却在诊断中看到结构缺失），只在 Step 3 问题摘要里如实列出，不得在 Step 6 顺手展开修改。

🔴 **CHECKPOINT——补全内容矛盾自检**：完成本步全部补全/丰富后，逐条检查新增或增强的内容之间、以及新增内容与原文之间是否存在矛盾（如两处业务规则对同一场景给出不同判定、新增验收标准与原有功能描述的范围不一致、新增交互细节暗示的前提与原有前置条件冲突）。发现矛盾时不得自行选边定案继续写下去，必须在 Step 7 的"优化说明"中列出该矛盾点并标注"⚠️矛盾待确认"，正文对应处保留分歧的两种表述并说明需要用户拍板。

**Step 7 — 输出优化后完整 PRD**
输出完整的优化版 PRD，后附"优化说明"：

```
优化说明
├── 新增内容：[列出新增的章节/子节]
├── 增强内容：[列出丰富了内容的部分]
├── 修正内容：[列出修正了逻辑/术语的部分]
├── 保持不变：[列出未做修改的部分及原因]
└── ⚠️矛盾待确认：[列出Step6自检发现但未能自行判定的矛盾点，需用户确认；若无矛盾则写"无"]
```

**Step 8 — 优化后 Review 对比**
对照 `references/prd-checklist.md` 重新评分，输出对比：

```
优化效果对比
| 维度 | 优化前 | 优化后 | 提升 |
|------|--------|--------|------|
| 结构完整性 | XX | XX | +XX |
| 内容详细度 | XX | XX | +XX |
| ...         | ...| ...| ... |
| 综合得分 | XX/100 | XX/100 | +XX |
```

## 异常处理

| 触发条件 | 处理方式 |
|---------|---------|
| references/ 下任一文件读取失败或不存在 | 🔴 **STOP**——不得凭记忆或常识继续优化，直接告知用户"prd-optimizer 所需的参考文件缺失，无法优化"，并指出具体缺失的文件名 |
| 用户提供的内容为空或明显过短（不足以对应7章任一实质内容） | 🔴 **STOP**——回复"内容过短，无法进行诊断和优化，请提供完整 PRD 文档路径或内容" |
| 提供的内容明显不是 PRD（如代码、竞品分析、纯需求口头描述等） | 🔴 **STOP**——告知"这似乎不是 PRD 文档，prd-optimizer 只优化结构化 PRD"，不得直接进入诊断和优化流程，除非用户明确回复确认按此内容继续 |
| 用户提供的文件路径不存在或无读取权限 | 🔴 **STOP**——告知具体错误信息，请用户确认路径是否正确，或改为直接粘贴文档内容 |
| Step 5 询问优化重点后用户未给出明确选项（如说"随便""你看着办"） | 🔴 **STOP**——不得默认选D全面优化后自行继续，回复"为避免优化方向偏离预期，请至少选择A/B/C/D中的一项，或说明你最关心的具体问题" |

## Red Flags

| 错误做法 | 正确做法 |
|---------|---------|
| 不询问优化重点直接全量修改 | 必须先询问优化重点再执行（Step 5） |
| 修改原文功能意图或删减原有内容 | 只做增强补全，原有内容必须保留 |
| 输出优化版本但不说明改了什么 | Step 7 中"优化说明"是必需的 |
| 跳过优化后 Review | Step 8 是强制步骤，用于量化优化效果 |
| 优化后仍存在空白占位符 | 所有章节必须有实质内容，不得保留"[说明]"等占位 |
| 优化时引入与原需求不符的内容 | 不确定的内容需注明"待确认"而非凭空创作 |
| 发现补全内容间/与原文矛盾时自行选边定案 | 必须按 Step 6 CHECKPOINT 停止选边，两种表述都保留并在优化说明标注"⚠️矛盾待确认" |
| 用户只选了部分优化重点（如仅C可执行性）却擅自扩大到其他重点（顺手把结构/内容详细度也改了） | 严格限定在用户选定的范围内执行，未被选中的重点即使有明显缺口也不得主动展开优化 |
