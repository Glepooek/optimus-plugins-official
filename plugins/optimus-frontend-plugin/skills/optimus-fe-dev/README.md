# optimus-fe-dev 用户使用手册

> **版本：v1.1.0** | 前端全流程开发技能

## 这是什么？

`optimus-fe-dev` 是一个 AI 驱动的前端开发技能，帮你完成从需求分析到代码生成、测试交付物输出的全流程。你只需要提供 4 份文档，它就能自动：

- 分析需求和设计稿
- 探测项目技术栈
- 生成符合项目风格的代码
- 自动集成埋点（神策 SDK）
- 输出标准化测试交付物

---

## 快速开始

### 第一步：准备 4 份文档

| # | 文档 | 说明 | 必填 |
|---|------|------|------|
| 1 | **PRD 需求文档** | 产品需求链接（飞书/Confluence 等） | 是 |
| 2 | **前端架构设计文档** | 前端架构方案链接（没有的话工具会自动生成） | 是 |
| 3 | **设计稿** | Figma 链接 或 Sketch 文件 | 是 |
| 4 | **API 接口文档** | Swagger 链接 或 Markdown 接口文档链接 | 是 |
| 5 | **业务组件库 Storybook** | 如果项目有业务组件库，提供 URL | 可选 |

### 第二步：告诉 AI 开始

在 Claude Code 中直接说：

```
帮我还原这个页面：
- PRD：https://xxx.feishu.cn/docx/xxx
- 设计稿：https://www.figma.com/design/xxx
- 接口文档：https://xxx.feishu.cn/docx/xxx
- 架构文档：https://xxx.feishu.cn/docx/xxx
```

如果你没有架构文档，也可以先让工具帮你生成：

```
根据这个 PRD 生成前端架构设计文档：https://xxx.feishu.cn/docx/xxx
```

### 第三步：确认方案，等待完成

AI 会分析所有文档后给你一份**实施计划**，包含文件清单和任务拆分。确认后自动生成代码和交付物。

## 工作流程（5 个阶段）

```
准备文档 → 分析规划 → 确认方案 → 生成代码 → 输出交付物 → 验证完成
```

### 阶段 1：收集输入

- 你提供 4 份文档链接
- 工具自动读取文档内容（飞书链接自动识别，无需特殊处理）
- 验证每份文档是否完整可用

### 阶段 2：分析与规划

- 自动探测项目技术栈（框架、语言、CSS 方案、UI 库等）
- 解析 PRD 提取页面、功能、交互、数据结构
- 解析设计稿提取布局、样式、组件
- 解析接口文档提取 API 接口和数据模型
- 生成实施计划：文件清单 + 任务拆分 + 依赖关系

### 阶段 3：代码生成

- 按任务依赖顺序逐个生成：类型定义 → API 函数 → 页面组件
- 每个任务自动双重 review：规格合规 + 代码质量
- 100% 还原设计稿（布局、间距、颜色、字体、圆角、阴影）
- 自动集成埋点代码

### 阶段 4：生成交付物

生成 5 份标准化测试文档：

| 交付物 | 内容 |
|--------|------|
| **功能说明书** | 功能列表、表单校验规则、状态枚举、权限说明 |
| **接口清单** | 接口概览、请求/响应示例、错误场景 |
| **UI 状态清单** | 所有页面状态（加载中、空数据、报错、正常等） |
| **Mock 数据** | 正常/空数据/异常场景的模拟数据 |
| **架构设计文档** | 技术栈、目录结构、组件拆分、数据流 |

### 阶段 5：验证完成

- 代码质量检查（风格、类型安全、导入）
- PRD 覆盖度检查（所有功能、字段、校验、权限）
- 设计稿还原度检查
- API 合规检查
- 交付物完整性检查

## 代码规范概要

工具生成的代码遵循 **9 层架构规范**：

| 层 | 说明 | 关键规则 |
|----|------|----------|
| 主程序 | `main.tsx` / `App.tsx` | 神策 SDK 在 App 渲染前初始化 |
| 全局样式 | `styles/globals.css` | Tailwind 指令 + CSS 变量 |
| 页面 | `pages/xxx/index.tsx` | 四段式结构：状态 → 数据 → 事件 → 渲染 |
| 组件 | `components/XxxName/index.tsx` | Props 必须类型化，Tailwind 优先 |
| 接口层 | `api/xxx.ts` | 一个业务模块一个文件，完整类型标注 |
| 数据层 | `stores/` + `hooks/` | Zustand 管客户端状态，TanStack Query 管服务端数据 |
| 类型层 | `types/xxx.ts` | 严格 TypeScript，禁止 `any` |
| 工具层 | `utils/xxx.ts` | 纯函数，无 React 依赖 |
| 埋点 | 集成在各层 | PV/点击/曝光/表单/错误 全覆盖 |

## 推荐技术栈

适用于新建项目：

| 层 | 技术 |
|----|------|
| 框架 | React 18+ |
| 语言 | TypeScript 5.x（strict 模式） |
| 样式 | Tailwind CSS 3+ |
| 客户端状态 | Zustand |
| 服务端状态 | TanStack Query v5 |
| 构建工具 | Vite 5+ |
| HTTP 客户端 | axios |
| UI 组件库 | Ant Design 5 / shadcn/ui |
| 路由 | React Router v6 |
| 包管理器 | pnpm |

> 注意：已有项目会自动探测技术栈，不会强制使用上述配置。

## 飞书文档支持

当文档链接包含 `feishu.cn` 或 `larksuite.com` 时，工具会自动使用飞书 API 读取内容。

**首次使用需登录：**

```bash
npx @larksuite/cli auth login
```

支持的操作：读取文档、上传文档、导出文档、搜索文档。

## 注意事项

1. **4 份文档缺一不可** — 在所有文档就绪前不会开始生成代码
2. **设计稿不能跳过** — 即使看起来复杂，MCP 工具能处理
3. **不要跳过确认步骤** — 阶段 2 生成的实施计划需要你确认后才会开始编码
4. **交付物是流程的一部分** — 不是可选的附加品
5. **代码风格跟随项目** — 工具会读取现有代码学习风格，生成的代码看起来像同一个人写的
