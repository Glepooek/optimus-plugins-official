# 驾驭 Claude 的智能：构建更好的 Agent 工具链

> **来源：** [Claude Blog - Harnessing Claude's Intelligence](https://claude.com/blog/harnessing-claudes-intelligence)
> **作者：** Lance Martin（Claude 平台团队）
> **发布日期：** 2026 年 4 月 2 日
> **阅读时长：** 5 分钟

---

Anthropic 联合创始人之一 Chris Olah 曾说，像 Claude 这样的生成式 AI 系统，与其说是被"构建"出来的，不如说是被"培育"出来的。研究人员塑造成长的条件，但涌现出的确切能力并不总是可以预测的。

这给基于 Claude 进行构建带来了一个挑战：Agent 工具链编码了关于 Claude 局限性的假设，但随着 Claude 不断进步，这些假设会逐渐过时。即便是已发布的指南，也值得频繁重新审视。

本文分享三种模式，帮助团队构建能够跟上 Claude 持续演进智能的应用，同时兼顾延迟和成本的平衡：

1. **使用 Claude 已经理解的工具**
2. **持续追问"我能停止做什么？"**
3. **谨慎地通过工具链设置边界**

---

## 一、使用 Claude 本身就理解的工具

我们建议使用 Claude 本身就理解得很好的工具来构建应用。具体来说，是 bash 工具和文本编辑器工具。

bash 并非为构建 Agent 而设计，但它是 Claude 知道如何使用、且随时间不断用得更好的工具。Claude Code 正是建立在这两个基础工具之上的。

到 2024 年底，Claude 3.5 Sonnet 仅凭 bash 工具和一个用于查看、创建、编辑文件的文本编辑器工具，在 SWE-bench Verified 上达到了 **49%** 的得分。

![SWE-bench 得分对比](./assets/harnessing-image2.png)

我们看到 Claude 将这些通用工具组合成能解决不同问题的模式。以下三种能力均由 bash 和文本编辑器工具组合而来：

- **Agent Skills（技能）**：将专业知识打包为按需加载的指令集
- **程序化工具调用（Programmatic Tool Calling）**：让 Claude 自主决定调用哪些工具及调用逻辑
- **记忆工具（Memory Tool）**：让 Claude 将重要信息持久化到文件

---

## 二、持续追问"我能停止做什么？"

Agent 工具链编码了关于 Claude 自身无法完成什么的假设。随着 Claude 变得更有能力，这些假设应该被持续检验。

这一原则涵盖三个维度：编排（Orchestration）、上下文管理（Context Management）和持久化（Persistence）。

### 让 Claude 自主编排动作

一个普遍的假设是：每个工具调用的结果都应该流经 Claude 的上下文窗口，以指导下一个动作。但将工具调用结果处理成 token 既慢、又昂贵，而且在结果只需要传递给下一个工具时完全没有必要。

设想 Claude 读取一张大表格，只为推断其中一列的值：整张表格都会进入上下文，Claude 为每一行无关数据付出 token 代价。虽然可以在工具设计层面用硬编码过滤器来解决这个问题，但这并没有解决根本矛盾：工具链正在做一个本应由 Claude 来做、且 Claude 更擅长做的编排决策。

给 Claude 一个代码执行工具可以解决这个问题——Claude 用代码来表达工具调用以及调用之间的逻辑，自行决定哪些结果需要传递、过滤或管道到下一次调用，而无需经过上下文窗口。只有代码执行的输出才会进入 Claude 的上下文窗口。编排决策从工具链转移到了模型本身。

因为代码是 Claude 编排动作的通用方式，一个强大的编程模型同时也是一个强大的通用 Agent。

在 BrowseComp 基准测试中，这一改进使 Opus 4.6 的准确率从 **45.3% 跃升至 61.6%**。

![BrowseComp 编排效果对比](./assets/harnessing-image4.png)

### 让 Claude 自主管理上下文

预先把所有任务特定的指令塞进系统提示词并不能很好地扩展：每增加一个 token，都在消耗 Claude 的注意力预算，而很少用到的指令只是在浪费资源。

**Skills** 通过渐进式披露解决这个问题：每个 Skill 的 YAML frontmatter 是预加载进上下文窗口的简短描述，提供该 Skill 内容的概览；完整的 Skill 内容只在 Claude 判断某项任务确实需要时，才通过调用读取文件工具按需加载。

**上下文编辑（Context Editing）** 是反向操作：提供了一种选择性移除已过期或不相关上下文的方式，例如旧的工具调用结果或思考过程块。

**子代理（Subagents）** 让 Claude 在需要时分叉出一个新的上下文窗口，以隔离特定任务的工作，避免污染主代理的上下文窗口。使用 Opus 4.6 时，这一能力使 BrowseComp 结果比最佳单 Agent 运行提升了 **2.8%**。

![上下文管理对比](./assets/harnessing-image7.png)

### 让 Claude 自主持久化上下文

长时运行的 Agent 可能会超出单个上下文窗口的限制。一个普遍的假设是：记忆系统应该依赖模型外围的检索基础设施。而我们大量的工作聚焦于为 Claude 提供简单的方式，让它自己选择要持久化哪些内容。

**Compaction（压缩）** 允许 Claude 在上下文接近限制时，自主将过去的对话历史总结为精华。这个过程不是机械截断，而是由模型决定什么值得保留。

**记忆文件夹（Memory Folder）** 是另一种方式，允许 Claude 将上下文写入文件，并在需要时读取。

Compaction 的效果随模型能力的提升而显著放大。值得注意的是，Sonnet 4.5 无论分配多少压缩预算，准确率都停滞在 43%，而 Opus 4.6 在相同配置下达到了 84%——这说明"选择什么值得保留"这种能力本身，会随模型能力的提升而扩展。

| 方案 | BrowseComp 准确率 |
|---|---|
| Sonnet 4.5 + Compaction | 43%（平台期，无法突破）|
| Opus 4.5 + Compaction | 68% |
| Opus 4.6 + Compaction | **84%** |
| Sonnet 4.5 + 记忆文件夹（BrowseComp-Plus）| 60.4% → **67.2%** |

![Compaction 效果图](./assets/harnessing-image6.png)

以宝可梦这类长周期游戏为例，可以清晰地看出记忆质量的差异。Sonnet 3.5 将记忆当作流水账，记录的是 NPC 说了什么，而不是什么真正重要。经过 14000 步后，它存了 31 个文件——包括两个关于毛毛虫宝可梦的近乎重复的笔记——而游戏进度仍停留在第二个城镇。相比之下，Opus 4.6 在同等步数时，已经有 10 个按目录组织的文件、3 枚道馆徽章，以及一个从自身失败经历中提炼出的学习笔记。

**Sonnet 3.5 的记忆（信息密度低）：**

```
caterpie_weedle_info:
- Caterpie and Weedle are both caterpillar Pokémon.
```

**Opus 4.6 的记忆（具有战术价值）：**

```
/gameplay/learnings.md:
- Bellsprout Sleep+Wrap combo: KO FAST with BITE before Sleep Powder lands.
- Gen 1 Bag Limit: 20 items max. Toss unneeded TMs before dungeons.
```

![记忆质量对比](./assets/harnessing-image3.png)

## 三、谨慎设置边界

并非所有工具链逻辑都是"多余的脚手架"。Agent 工具链为 Claude 提供了用于约束用户体验、控制成本或保障安全的结构。关键在于**区分哪些边界仍然必要，哪些已经成为死重**。

### 设计上下文以最大化缓存命中率

Messages API 是无状态的——Claude 无法看到之前的对话历史，因此工具链每一轮都必须重新打包所有过去的动作、工具描述和指令。

提示词通过设置断点进行缓存：API 将断点之前的上下文写入缓存，并检查该上下文是否与之前的缓存条目匹配。缓存命中的 token 成本仅为基础输入 token 的 **10%**。

要最大化缓存命中率，需要遵循以下原则：

| 原则 | 说明 |
|---|---|
| **静态内容优先，动态内容置后** | 将系统提示词、工具定义等稳定内容放在前面 |
| **用消息追加更新，而非编辑提示词** | 在消息中追加 `<system-reminder>` 而非修改提示词本身，避免破坏缓存 |
| **不要在会话中切换模型** | 缓存是模型特定的，切换模型会导致缓存失效；如需切换，改用子代理 |
| **谨慎管理工具列表** | 工具位于缓存前缀中，增删任何一个工具都会使整个前缀缓存失效；使用工具搜索实现动态发现 |
| **及时移动缓存断点** | 将断点移至最新消息位置；多轮应用使用自动缓存功能 |

![缓存结构示意图](./assets/harnessing-image5.png)

### 使用专用工具划定 UX、可观测性或安全边界

Claude 发出工具调用，由工具链处理。bash 工具赋予 Claude 广泛的编程操控能力，但它给工具链的只是一个命令字符串——每个动作都是同样的形状。将操作提升为专用工具，工具链就能获得动作专用的钩子，带有类型化的参数，可以被拦截、审批、渲染或审计。

难以撤销的操作（如外部 API 调用）是专用工具的天然候选——可以设置用户确认门控。写入类工具可以内置过期检查，避免 Claude 覆盖自上次读取后已发生变更的文件。

专用工具也服务于展示层：以弹窗形式渲染，向用户展示问题、提供选项，或在等待用户反馈时阻塞 Agent 循环。

对于可观测性而言，类型化的工具调用能让工具链获得结构化参数，支持日志记录、链路追踪和回放。

这一决定应该被持续重新评估。随着 Claude 能力的提升，曾经需要通过工具链强制执行的限制，可能已经可以依靠模型自身的判断来实现。Claude Code 的 **auto-mode** 就是一个例子：在 bash 工具周围建立安全边界，由第二个 Claude 实例读取命令字符串，判断其是否安全。这种模式可以减少对专用工具的依赖，但仅适用于用户信任 Agent 总体方向的任务场景。即便如此，高风险操作仍可能需要专用工具。

![专用工具边界示意图](./assets/harnessing-image1.png)

## 结语

Claude 智能的前沿始终在变化。关于 Claude 无法完成什么的假设，需要在每一次能力跃升时重新检验。

我们反复看到这一规律：在使用 Sonnet 4.5 进行长任务时，团队发现模型会在感知到上下文窗口接近上限时提前放弃任务——即使任务还远未完成。为此，工程团队构建了一套上下文重置机制，在达到阈值时清空上下文窗口。然而，当 Opus 4.5 发布后，这一行为完全消失了，为此构建的上下文重置机制变成了工具链中真正的"死重"。

移除这些死重非常重要，因为它们会成为 Claude 性能的瓶颈。随着时间推移，我们应用中的结构和边界，都应该基于这个问题来持续剪枝：**我能停止做什么？**
