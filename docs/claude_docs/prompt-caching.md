# Claude Code 如何使用提示词缓存

> Claude Code 自动管理提示词缓存。本文说明为何切换模型会触发一次较慢的未命中缓存轮次、`/compact` 的代价是什么、为何 CLAUDE.md 的编辑在会话中途不生效，以及如何检查缓存命中率。

来源：[code.claude.com/docs/en/prompt-caching](https://code.claude.com/docs/en/prompt-caching)

---

提示词缓存使 Claude Code 更快、更省钱。没有缓存时，API 每一轮都需要重新处理完整的历史记录。有了缓存，它可以复用已处理的内容，只对变化的部分做新工作。

Claude Code 会自动处理提示词缓存，除非你[禁用它](#禁用提示词缓存)。不过了解提示词缓存的工作方式仍然有用，因为某些操作会使缓存失效，导致下一次响应在重建缓存时变慢且更昂贵。本文介绍哪些操作会失效、某些设置为何要等到重启才生效，以及当用量看起来偏高时如何检查缓存性能。

---

## 缓存的组织方式

每次在 Claude Code 中发送消息，都会发起一个新的 API 请求。模型在请求之间不保留任何记忆，因此 Claude Code 会重新发送完整的上下文：系统提示词、项目上下文、所有先前的消息和工具调用结果，以及你的新消息。新内容追加在末尾，这意味着每次请求的大部分内容与上次相同。提示词缓存就是 API 避免重复处理未变化部分的机制。

API 通过将请求开头（称为**前缀**）与最近处理过的内容进行匹配来实现缓存。正常轮次下，前缀就是整个上一次请求，只有最新的对话是新内容。匹配是精确的，因此前缀中任何位置的变化都会导致其后所有内容重新计算。不存在按文件或按片段的缓存。底层机制参见 API 参考中的[提示词缓存工作原理](https://platform.claude.com/docs/en/build-with-claude/prompt-caching#how-prompt-caching-works)。

![四轮对话的前缀缓存示意图：每轮请求包含前一轮的全部内容加上最新对话追加在末尾。第二、三轮中，未变化的前缀从缓存读取，只有新增内容被处理。第四轮中系统提示词发生变化，前缀不再匹配，整个请求被重新处理并写入缓存。](assets/prompt-caching-prefix.svg)

为最大化前缀匹配效果，Claude Code 对每次请求按内容稳定性排序，变化频率低的内容排在前面：

| 层级 | 内容 | 何时变化 |
|------|------|---------|
| 系统提示词 | 核心指令、工具定义、输出风格 | 加载的工具定义集发生变化，或 Claude Code 升级时 |
| 项目上下文 | CLAUDE.md、自动记忆、无作用域规则 | 会话启动时，或执行 `/clear` 或 `/compact` 后 |
| 对话内容 | 你的消息、Claude 的回复、工具调用结果 | 每轮都变化 |

对话层的变化不影响系统提示词和项目上下文的缓存。系统提示词的变化则会使所有内容失效，因为后续所有内容都位于不同的前缀之后。第三列列出的是常见触发因素而非完整列表，下文各节涵盖完整集合，包括输出风格等在会话启动时固定的内容。

计划模式（plan mode）和 skill 加载会将其指令追加为对话消息，因此缓存前缀保持完整。

有两个设置不属于提示词文本，但都是缓存键的一部分：

- **模型**：每个模型有自己的缓存。切换模型会重新计算整个请求，即使内容完全相同。
- **努力级别（effort level）**：同一模型的每个努力级别有自己的缓存。中途更改会重新计算整个请求，Claude Code 会在应用此更改前要求你确认。

> 💡 在会话开始时选好模型和努力级别，将 `/compact` 保留用于任务间的自然间隔。中途改动越少，缓存命中率越高。

### 缓存存储在哪里

缓存发生在服务端，位于为你的模型提供服务的基础设施中。具体位置取决于你的认证方式：

- **API key、Claude 订阅或 Claude Platform on AWS**：缓存在 Anthropic 的基础设施中
- **Bedrock 或 Vertex AI**：缓存在你的云服务商的服务基础设施中
- **Foundry**：请求路由到 Anthropic 的基础设施
- **自定义 `ANTHROPIC_BASE_URL` 或 LLM 网关**：缓存位于你的请求转发目标，是否生效取决于网关

各提供商存储和处理的内容详见[数据使用说明](https://code.claude.com/docs/en/data-usage)。无论缓存在哪里，条目都会在不活跃一段时间后过期，[缓存生命周期](#缓存生命周期)一节介绍 TTL 及其延长方式。

## 使缓存失效的操作

这些操作会导致下一次请求部分或全部未命中缓存。你会看到一次较慢、较贵的轮次，之后新前缀就被缓存了。大多数操作在了解其代价后是可以避免在任务进行中执行的。切换模型感觉像是免费的，直到你注意到紧随其后的那次较慢轮次。

### 切换模型

每个模型有自己的缓存。使用 `/model` 切换模型意味着下一次请求读取整个对话历史时不会有缓存命中，即使内容完全相同。

`opusplan` 模型设置在计划模式下解析为 Opus，在执行模式下解析为 Sonnet，因此每次进出计划模式都是一次模型切换，会重新建立缓存。

Fable 5 上的自动模型回退也是一次模型切换——当安全分类器标记某请求时，Claude Code 会在默认 Opus 模型上重新运行。

### 更改努力级别

缓存同时以努力级别和模型为键，因此使用 `/effort` 切换后，下一次请求读取整个对话历史时不会有缓存命中。一旦对话已开始，在会话中途更改努力级别，Claude Code 会显示确认对话框。如果更改解析后与当前生效的级别相同（例如显式设置模型的默认级别），则跳过对话框并保持缓存。

### 开启快速模式

启用快速模式会添加一个作为缓存键一部分的请求头，因此下一次请求会整体未命中缓存。这也是为什么在会话开始时开启快速模式的代价低于在长会话深处才开启的原因——未缓存的 input token 按快速模式费率计费。从非 Opus 模型启用快速模式还会[切换模型](#切换模型)，后者本身就会重新建立缓存。

开启后，Claude Code 在后续请求中保持发送该请求头，且只改变请求的速度设置（不属于缓存键）。关闭快速模式、因速率限制自动回退到标准速度、以及再次开启，都不会影响缓存。

> 注意：跨切换保持请求头需要 Claude Code v2.1.86 或更高版本。旧版本中每次快速模式切换和速率限制回退都会使缓存失效。

### 连接或断开 MCP 服务器

工具定义位于系统提示词层，因此当请求中的工具定义集在两次请求之间发生变化时，缓存就会失效。advisor 工具是例外：其定义位于缓存断点之后，因此启用或禁用 `/advisor` 不会影响缓存前缀。

MCP 服务器的变化是否影响缓存，取决于其工具是被工具搜索（tool search）延迟加载还是直接加载进前缀：

- **延迟工具**（支持的模型上的默认行为）：服务器连接、断开或更改工具列表只会追加新内容，不会影响已缓存的内容。
- **加载进前缀的工具**：对它们的任何更改都会使缓存失效。这发生在工具搜索不可用或被禁用时（如 Haiku 模型、Vertex AI 或自定义 `ANTHROPIC_BASE_URL` 网关），以及标记了 `alwaysLoad` 的服务器或工具，以及通过阈值加载保持在前端的定义。

工具加载进前缀时，最常见的失效原因是会话中途某个服务器连接或断开——这可能在你没有任何操作的情况下发生：stdio 服务器进程退出、HTTP 会话过期，或者服务器在瞬时故障后自动重连。已连接的服务器也可能推送动态工具更新来改变其工具列表。

编辑 MCP 配置本身不会改变缓存，新配置只有在重启后服务器连接或断开时才生效。

### 启用或禁用 plugin

plugin 捆绑了多种组件类型，变化的代价取决于 plugin 提供的组件。Skills、commands、agents、hooks、LSP 服务器、monitors 和主题不会使缓存失效：它们添加到请求中的任何内容都追加在现有对话之后。

例外是提供 MCP 服务器的 plugin——遵循与连接或断开 MCP 服务器相同的规则。

Plugin 变更在运行 `/reload-plugins` 或开始新会话时生效。从 v2.1.163 起，当重新加载会触发完整重读时，`/reload-plugins` 会显示警告并不应用重载，可传入 `--force` 强制应用。

在会话中途禁用一个早先已启用的 plugin，会恢复之前的请求形态。如果该前缀仍在其[缓存生命周期](#缓存生命周期)内，下一次请求将读取旧缓存条目而非重建。

### 拒绝整个工具

将 `Bash` 或 `WebFetch` 这样的裸工具名称添加为拒绝规则会从 Claude 的上下文中完全移除该工具。内置工具定义加载在系统提示词层，因此会话中途添加或移除这类规则会使缓存失效。

只有在工具名称位置匹配的拒绝规则才有此效果：裸工具名、等效的 `Bash(*)` 形式，或 `"*"` 这样的工具名通配符。范围限定的拒绝规则（如 `Bash(rm *)`）以及所有允许和询问规则，不改变 Claude 看到的工具，不影响前缀。

### 压缩对话

压缩（compaction）将消息历史替换为摘要，因此从设计上就会使对话层失效。Claude Code 重用系统提示词层并从磁盘重新加载项目上下文，只有当 CLAUDE.md 和记忆自会话开始以来未变化时才会命中缓存。

生成摘要的请求会复用你的前缀，因此该请求读取现有缓存而非重新处理完整历史。压缩的大部分时间花在生成摘要上，而非缓存未命中。压缩后的下一轮只需为更短的摘要重建对话缓存。

> 💡 当你丢弃的上下文是不再需要的内容时，压缩对你有利。在任务之间的自然间隔（而非任务进行中）运行 `/compact` 来控制其开销。如果你已经走入了一条想完全放弃的路径，改用 `/rewind` 回到更早的轮次——这会截断到一个已缓存的前缀，而非像压缩那样建立新前缀。

### 升级 Claude Code

新版本通常会更新系统提示词或工具定义，因此升级后的第一次请求会从头重建缓存。自动更新在后台下载新版本，但在下次启动时才应用，因此你会看到重启后的第一轮未命中缓存，而不是在会话中途意外发生。设置 `DISABLE_AUTOUPDATER=1` 可控制升级时机。

> 注意：升级后恢复会话（resume a session）会重新处理整个对话历史，没有任何缓存命中，因为历史现在位于不同系统提示词之后。代价与恢复的对话长度成正比，因此回到长会话的第一轮可能是你发送的最贵的请求。

## 不影响缓存的操作

这些操作要么追加到对话末尾，要么完全不触及请求。其中一些（如编辑 CLAUDE.md 或更改输出风格）也解释了为何设置变更需要等到重启才生效。

### 编辑仓库中的文件

文件内容只在 Claude 读取时才进入上下文，读取会追加到对话。编辑 Claude 之前读取过的文件，不会改变历史中的早期读取记录。Claude Code 会追加一条 `<system-reminder>` 说明文件已更改，Claude 会在需要时重新读取。

### 会话中途编辑 CLAUDE.md

项目根目录和用户级别的 CLAUDE.md 在会话启动时读取一次并保存在内存中。会话中途编辑不会使缓存失效，但编辑也不会生效——Claude 继续使用会话启动时加载的版本。新内容在下次 `/clear`、`/compact` 或重启时加载。

子目录中的嵌套 CLAUDE.md 和带 `paths:` frontmatter 的规则在 Claude 首次读取匹配文件时才加载。在其加载前编辑会生效；加载后内容已是对话历史的一部分，会话中途的编辑不会追溯更改它。

### 更改输出风格

输出风格是系统提示词的一部分，在会话启动时读取一次。通过 `/config` 或 `outputStyle` 设置在会话中途更改，不会使缓存失效，但更改也不会生效。新风格在下次 `/clear` 或重启时加载。

### 更改权限模式

在权限模式之间切换（如从默认模式切换到接受编辑模式）不会更改系统提示词或工具定义，因此是缓存安全的。例外是使用 `opusplan` 模型设置时的计划模式——进出计划模式时会在 Opus 和 Sonnet 之间切换模型，即构成一次[模型切换](#切换模型)。

### 调用 skills 和 commands

Skills 和 commands 在调用点将其指令注入为用户消息，对话中更早的内容不受影响。

### 运行 `/recap`

`/recap` 生成摘要显示在终端中。与 `/compact` 不同，它将摘要追加为命令输出而非替换消息历史，因此缓存前缀保持完整。

### 回退对话（/rewind）

`/rewind` 将对话截断回更早的轮次。剩余的历史是缓存在那时建立的同一内容，因此下一次请求命中更早的缓存条目。由于每一轮都在读取该前缀，即使原始轮次比 TTL 更早，条目也保持热态。

随对话一起恢复文件检查点对缓存没有单独影响。文件内容只在 Claude 读取时才进入上下文，与[编辑仓库中的文件](#编辑仓库中的文件)的情况相同。

## 缓存生命周期

缓存前缀在一段不活跃时间后过期。每次命中缓存的请求都会重置计时器，因此只要你持续工作，缓存就会保持热态。停顿时间足够长后，下一次请求会重新计算完整输入并重建缓存——这就是暂离一段时间后第一轮回来可能明显变慢的原因。

TTL 控制缓存能存活多长时间的间隔。API 提供两种：五分钟 TTL 和一小时 TTL（后者在较长休息后保持缓存热态，但缓存写入以更高费率计费）。Claude Code 根据你的认证方式为你选择 TTL，你也可以通过环境变量覆盖。

### Claude 订阅用户

使用 Claude 订阅时，Claude Code 自动请求一小时 TTL。使用量包含在你的套餐中而非按 token 计费，因此更长的 TTL 不会产生额外费用，只会影响缓存保持热态的时长。

如果你已超出套餐使用上限并使用额外用量积分，Claude Code 会自动将 TTL 降为五分钟。

### API key 或第三方提供商用户

使用 API key、Bedrock、Vertex、Foundry 或 Claude Platform on AWS 时，你按 token 费率付费，因此 TTL 默认为更便宜的五分钟。若要使用一小时 TTL，设置 `ENABLE_PROMPT_CACHING_1H=1`。

在 Bedrock 上，提示词缓存支持、最小可缓存前缀长度和一小时 TTL 可用性因模型而异。如果缓存 token 数量始终为零，请查看 Bedrock 文档中的支持模型、区域和限制。

### 覆盖 TTL

设置 `FORCE_PROMPT_CACHING_5M=1` 可强制使用五分钟 TTL，无论认证方式如何。这在调试缓存行为、比较两种 TTL 或覆盖托管设置中的 `ENABLE_PROMPT_CACHING_1H` 时很有用。

## 缓存作用域

在 Claude Code 中，缓存实际上作用于一台机器和一个目录。系统提示词嵌入了工作目录、平台、shell、OS 版本和自动记忆路径，因此不同目录中的两个会话会建立不同的前缀，无法互相命中缓存——包括同一仓库的不同 worktree，因为每个 worktree 有自己的工作目录。

在同一目录中并行运行的会话会建立匹配的前缀并读取彼此的缓存。顺序会话只有在启动时的 git 状态快照匹配时才共享前缀（系统提示词也会捕获分支和最近提交）。

底层 API 缓存的范围更广——在组织之间（在某些提供商中，在组织内的工作区之间）隔离。在这些边界内，具有相同模型和前缀的任何两个请求读取同一缓存。对于运行自动化流程队列的 Agent SDK 调用者，参见[跨用户和机器改善提示词缓存](https://code.claude.com/docs/en/agent-sdk/modifying-system-prompts#improve-prompt-caching-across-users-and-machines)，了解如何屏蔽系统提示词中的按机器差异部分，实现跨机器共享缓存。

## 检查缓存性能

API 在每次响应中报告两个 token 计数。实时查看的最直接方式是读取 `current_usage` 对象的[状态栏脚本](https://code.claude.com/docs/en/statusline)：

| 字段 | 含义 |
|------|------|
| `cache_creation_input_tokens` | 本轮写入缓存的 token 数，按缓存写入费率计费 |
| `cache_read_input_tokens` | 本轮从缓存读取的 token 数，约为标准输入费率的 10% |

高读写比意味着缓存效果好。如果创建量逐轮持续偏高，说明前缀中有某些内容在变化，参见[使缓存失效的操作](#使缓存失效的操作)章节。

如需跨组织范围查看缓存情况，OpenTelemetry 导出器会按用户和会话上报缓存读取和创建的 token 数。指标和事件属性参考详见[监控用量](https://code.claude.com/docs/en/monitoring-usage)。

## 子智能体与缓存

子智能体（subagent）从自己的系统提示词和工具集开始独立对话，与父级分离。它建立自己的缓存，第一次调用时没有缓存命中，随后在自己的轮次中逐渐预热。子智能体使用五分钟 TTL，即使在订阅计划下也是如此。

父级的缓存不受影响——从父级的角度看，子智能体的调用和结果追加到对话中，父级前缀保持完整。

相比之下，**fork** 会精确继承父级的系统提示词、工具和对话历史，因此其第一次请求就能读取父级的缓存。压缩的摘要生成调用也使用相同的前缀共享方式。

## 禁用提示词缓存

在调试特定模型或提供商的缓存行为时，偶尔有禁用缓存的需求。设置以下任一环境变量为 `1` 即可关闭：

| 变量 | 效果 |
|------|------|
| `DISABLE_PROMPT_CACHING` | 对所有模型禁用 |
| `DISABLE_PROMPT_CACHING_HAIKU` | 仅对 Haiku 禁用 |
| `DISABLE_PROMPT_CACHING_SONNET` | 仅对 Sonnet 禁用 |
| `DISABLE_PROMPT_CACHING_OPUS` | 仅对 Opus 禁用 |
| `DISABLE_PROMPT_CACHING_FABLE` | 仅对 Fable 禁用 |

若要在组织范围内设置缓存策略，可将上述任一变量或 [TTL 变量](#缓存生命周期)放入[托管设置](https://code.claude.com/docs/en/settings#settings-files)的 `env` 块中。正常使用时保持缓存启用即可。
