最成功的 Claude Code 部署，在配置、工具链和组织结构上都呈现出可识别的共同模式。本文是《Claude Code 规模化》系列的一部分，该系列专门介绍在企业级规模使用 Claude Code 进行工程开发的最佳实践。

---

Claude Code 正在以下生产环境中运行：数百万行代码的 Monorepo、历经数十年的遗留系统、跨越数十个代码仓库的分布式架构，以及拥有数千名开发者的大型组织。这些环境带来了小型、简单代码库所没有的挑战——例如每个子目录的构建命令各不相同，或者遗留代码分散在没有共同根目录的文件夹中。

本文介绍了我们观察到的、能够推动 Claude Code 规模化成功落地的模式。"大型代码库"涵盖多种部署形态：数百万行代码的 Monorepo、历经数十年的遗留系统、跨多个独立仓库的数十个微服务，或以上各种形态的组合。还包括那些团队通常不会与 AI 编码工具联系在一起的语言场景，如 C、C++、C#、Java、PHP（尤其是在近期模型版本发布后，Claude Code 在这些语言上的表现优于大多数团队的预期）。虽然每个大型代码库的部署都受到其特定版本控制系统、团队结构和积累的约定规范的影响，但本文介绍的这些模式可以在不同场景下通用，是考虑采用 Claude Code 的团队的良好起点。

## Claude Code 如何导航大型代码库

Claude Code 像软件工程师一样导航代码库：遍历文件系统、读取文件、使用 grep 精确找到所需内容，并跟踪代码库中的交叉引用。它在开发者的本地机器上运行，不需要构建、维护或上传到服务器的代码库索引。

基于 RAG 的 AI 编码工具通过对整个代码库进行向量嵌入，在查询时检索相关片段来工作。在大规模场景下，这些系统可能失效——因为嵌入管道跟不上活跃工程团队的开发节奏。等到开发者查询索引时，它反映的可能是几周、几天甚至几小时前的代码库状态。检索结果随后可能返回两周前团队已重命名的函数，或引用上个迭代中已删除的模块，且没有任何提示说明这些信息已过时。

代理式搜索（Agentic Search）避免了上述失效模式。随着数千名工程师不断提交新代码，不需要维护嵌入管道或集中式索引。每位开发者的实例直接基于实时代码库工作。

但这种方式有一个权衡：当 Claude 有足够的起始上下文、知道去哪里查找时，效果最佳。这意味着 Claude 导航质量的高低，取决于代码库的配置质量——通过 CLAUDE.md 文件和 Skills 进行分层上下文配置。如果你让它在一个十亿行的代码库中查找某个模糊的模式，在工作开始之前就会触及上下文窗口限制。那些投入时间做好代码库配置的团队会获得更好的结果。

## 工具链（Harness）与模型同样重要

关于 Claude Code，最常见的误解之一是：其能力完全由所使用的模型决定。团队往往关注模型的基准测试结果以及它在测试任务上的表现。但在实践中，**围绕模型构建的生态系统——工具链——比模型本身更能决定 Claude Code 的实际表现**。

工具链由五大扩展点构成——CLAUDE.md 文件、Hooks、Skills、Plugins 和 MCP Servers——每个都承担不同的职责。团队构建它们的顺序很重要，因为每一层都建立在前一层的基础之上。另外两种能力——LSP 集成和子代理（Subagents）——构成了完整的配置体系。下面逐一解释这些组件和能力的作用：

### CLAUDE.md 文件是第一步

这些是 Claude 在每次会话开始时自动读取的上下文文件：根目录文件提供全局大局，子目录文件提供局部约定规范。它们为 Claude 提供了做好任何工作所需的代码库知识。由于这些文件在每次会话中都会加载，不论任务是什么，因此保持内容精准聚焦于广泛适用的内容，可以防止它们成为拖累性能的负担。

### Hooks 让配置能够自我改进

大多数团队将 Hooks 视为防止 Claude 犯错的脚本，但它们更有价值的用途是**持续改进**。Stop Hook 可以在会话结束时趁上下文还新鲜，反思本次会话发生的事情并提议更新 CLAUDE.md。Start Hook 可以动态加载团队特定的上下文，让每位开发者无需手动配置，就能在自己负责的模块上获得正确的配置。对于 Lint 和代码格式化等自动化检查，Hooks 以确定性的方式强制执行规则，比依赖 Claude 记住某条指令产生更一致的结果。

### Skills 让合适的专业知识按需可用，而不会让每次会话臃肿

在一个拥有数十种任务类型的大型代码库中，并非所有专业知识都需要在每次会话中存在。Skills 通过**渐进式披露**解决这个问题——将本会争夺上下文空间的专业工作流和领域知识卸载出去，仅在任务需要时加载。例如，安全审查 Skill 在 Claude 评估代码漏洞时加载，而文档处理 Skill 在代码变更后需要更新文档时加载。

Skills 还可以限定作用范围到特定路径，使其只在代码库的相关部分激活。负责支付服务的团队可以将他们的部署 Skill 绑定到该目录，这样在 Monorepo 其他地方工作时它永远不会自动加载。

### Plugins 分发有效的配置

大型代码库面临的挑战之一是，优秀的配置可能停留在**部落知识**层面。Plugin 将 Skills、Hooks 和 MCP 配置打包成单一可安装的包，这样当新工程师在第一天安装该 Plugin 时，就能立刻拥有与已经使用了一段时间的同事相同的上下文和能力。Plugin 更新可以通过受管理的 Marketplace 在组织内分发。

例如，我们合作的一家大型零售企业构建了一个 Skill，将 Claude 连接到他们的内部分析平台，让业务分析师无需离开工作流就能提取业绩数据。他们在大规模推广之前，先将其作为 Plugin 进行分发。

### 语言服务器协议（LSP）集成赋予 Claude 与开发者在 IDE 中相同的导航能力

大多数大型代码库的 IDE 中已经运行着 LSP，支撑着"跳转到定义"和"查找所有引用"功能。将 LSP 能力暴露给 Claude，赋予其符号级别的精确性：它可以追踪函数调用到其定义处，跨文件追踪引用，并区分不同语言中同名的函数。没有 LSP，Claude 依赖文本模式匹配，可能会定位到错误的符号。我们合作的一家企业软件公司，在 Claude Code 推广之前，特意在全组织范围内部署了 LSP 集成，以使 C 和 C++ 的导航在规模化场景下可靠运行。对于多语言代码库，这是价值最高的投资之一。

### MCP Servers 扩展一切能力

MCP Servers 是 Claude 连接到其他无法直接访问的内部工具、数据源和 API 的方式。最先进的团队构建 MCP Servers，将结构化搜索作为 Claude 可以直接调用的工具暴露出来。其他团队将 Claude 连接到内部文档、工单系统或分析平台。

### 子代理（Subagents）将探索与编辑分离

子代理是一个独立的 Claude 实例，拥有自己的上下文窗口，接受一个任务、完成工作，然后只将最终结果返回给父代理。一旦工具链就位，一些团队会启动一个只读的子代理来梳理某个子系统，将发现写入文件，然后由主代理带着完整的信息进行编辑。

| 组件                    | 是什么                             | 何时加载             | 最适合                               | 常见误区                                    |
| ----------------------- | ---------------------------------- | -------------------- | ------------------------------------ | ------------------------------------------- |
| CLAUDE.md               | Claude 自动读取的上下文文件        | 每次会话             | 项目特定约定、代码库知识             | 将应该放在 Skill 中的可复用专业知识放在这里 |
| Hooks                   | 在关键时刻运行的脚本               | 事件触发             | 自动化一致行为、捕获会话学习         | 对本应自动运行的事情使用提示词              |
| Skills                  | 为特定任务类型打包的指令           | 按需加载，相关时触发 | 跨会话和项目的可复用专业知识         | 将所有内容加载进 CLAUDE.md                  |
| Plugins                 | 捆绑了 Skills、Hooks、MCP 配置的包 | 配置后始终可用       | 在组织内分发有效配置                 | 让优秀配置停留在部落知识层面                |
| LSP（语言服务器协议）\* | 通过语言专用服务器提供实时代码智能 | 配置后始终可用       | 符号级导航和类型语言中的自动错误检测 | 以为它是自动启用的                          |

## 成功部署中的三大配置模式

如何为大型代码库配置 Claude Code，在很大程度上取决于该代码库的结构。但我们在观察到的部署中，有三种模式始终出现。

### 模式一：让代码库在规模化场景下可导航

Claude 在大型代码库中的帮助能力，受限于其找到正确上下文的能力。每次会话加载过多上下文会降低性能，加载过少则让 Claude 盲目导航。最有效的部署在前期投入，让代码库对 Claude 可读。以下几种模式频繁出现：

- **保持 CLAUDE.md 文件精简且分层。** Claude 在遍历代码库时叠加式地加载它们：根目录文件提供大局，子目录文件提供局部约定。根目录文件应该只包含指引和关键注意事项；其他内容都会变成噪音。

- **在子目录中初始化，而非在仓库根目录。** 当 Claude 的范围限定在任务实际相关的代码库部分时，效果最好。在 Monorepo 中，这可能感觉反直觉，因为工具通常假设从根目录访问，但 Claude 会自动向上遍历目录树并加载沿途找到的每个 CLAUDE.md 文件，所以根目录层面的上下文永远不会丢失。

- **按子目录限定测试和 Lint 命令的范围。** 当 Claude 只修改了一个服务时，运行完整的测试套件会导致超时，并将无关输出浪费在上下文空间上。子目录层面的 CLAUDE.md 文件应该指定适用于该代码库部分的命令。这对面向服务的代码库效果很好，每个目录有自己的测试和构建命令。在有深层跨目录依赖的编译型语言 Monorepo 中，按子目录限定范围更难实现，可能需要特定于项目的构建配置。

- **使用 `.ignore` 文件排除生成文件、构建产物和第三方代码。** 在 `.claude/settings.json` 中提交 `permissions.deny` 规则意味着这些排除项是版本控制的，因此团队中的每位开发者都无需自己配置就能获得相同的噪音过滤。在某些代码库中，生成的文件本身就是开发工作的对象。从事代码生成器开发的开发者可以在本地设置中覆盖项目级别的排除项，而不影响团队其他人。

- **当目录结构无法完成导航工作时，构建代码库地图。** 对于代码不按照常规目录结构组织的组织，在仓库根目录放置一个轻量级 Markdown 文件，列出每个顶级文件夹及其一行简介，为 Claude 提供一个可以在打开文件前扫描的目录。对于有数百个顶级文件夹的代码库，这最好采用分层方式：根目录文件只描述最顶层的结构，子目录 CLAUDE.md 文件提供下一层细节，在 Claude 遍历树时按需加载。对于更简单的情况，用 `@` 提及 Claude 应该参考的特定文件或目录也能达到同样的效果。

- **运行 LSP 服务器，让 Claude 按符号而非字符串进行搜索。** 在大型代码库中 grep 一个常见函数名会返回数千个匹配项，Claude 为了弄清哪个重要而打开文件，会消耗大量上下文。LSP 只返回指向同一符号的引用，因此过滤在 Claude 读取任何内容之前就已完成。配置 LSP 需要为你的语言安装代码智能插件以及相应的语言服务器二进制文件；Claude Code 文档涵盖了可用的插件和故障排查说明。

**一个注意事项**：在某些极端情况下，即使是层级式 CLAUDE.md 方法也会失效——例如拥有数十万个文件夹和数百万个文件的代码库，或使用非 Git 版本控制的遗留系统。我们将在本系列的后续文章中专门讨论这些挑战。

---

### 模式二：随模型智能演进，主动维护 CLAUDE.md 文件

随着模型演进，为当前模型编写的指令可能会对未来的模型产生反作用。帮助 Claude 克服过去某些困难模式的 CLAUDE.md 文件，在下一个模型版本发布后，可能变得不再必要，甚至会造成主动限制。例如，一条告诉 Claude 将每次重构都拆分为单文件变更的规则，可能帮助早期模型保持正轨，但会阻止更新的模型进行它能够很好处理的协调性跨文件编辑。

为了弥补特定模型局限性——无论是模型推理能力还是 Claude Code 自身工具的局限——而构建的 Skills 和 Hooks，一旦这些局限不再存在，就会成为额外的开销。例如，一个拦截文件写入以在 Perforce 代码库中强制执行 `p4 edit` 的 Hook，在 Claude Code 添加了原生 Perforce 模式后就变得多余了。

团队应预期每三到六个月进行一次有意义的配置审查，同时在主要模型版本发布后性能感觉停滞不前时，也值得进行一次审查。

---

### 模式三：为 Claude Code 管理和推广明确归属所有权

仅凭技术配置无法驱动落地采用。那些做对了的组织，也在**组织层面**进行了投入。

推广最快的落地案例，都在开放广泛访问之前进行了专项基础设施投入。一个小团队——有时甚至只是一个人——提前将工具链配置好，让 Claude 在开发者第一次接触时就已经融入了他们的工作流程。在某家公司，几位工程师构建了一套 Plugins 和 MCP，在第一天就已可用。在另一家公司，一整个专注于管理 AI 编码工具的团队，在推广开始之前就已就位。在这两种情况下，开发者的第一次体验是高效的而非令人沮丧的，随后采用率从这里蔓延开来。

当前承担这项工作的团队，往往隶属于开发者体验（Developer Experience）或开发者生产力（Developer Productivity）部门，这通常也是负责新工程师入职和构建开发者工具链的职能部门。在多个组织中，一个正在兴起的角色是 **"Agent 管理员"**：一个专门管理 Claude Code 生态系统的 PM/工程师混合职能。对于没有专职团队的组织，最小可行方案是设置一名 DRI（直接负责人）：一个拥有 Claude Code 配置所有权的人，对配置设置、权限策略、Plugin Marketplace 和 CLAUDE.md 约定规范具有决策权，并负责保持它们的时效性。

自下而上的采用会产生热情，但如果没有人来集中整合有效的内容，就会碎片化。你需要有个人或团队来整合和推广正确的 Claude Code 约定（例如标准化的 CLAUDE.md 层级结构，或一套经过筛选的 Skills 和 Plugins）。没有这项工作，知识将停留在部落层面，采用率也将停滞不前。

在大型组织中，尤其是受监管行业的组织，治理问题会很早出现——例如：谁控制哪些 Skills 和 Plugins 可以使用，如何防止数千名工程师各自独立重建同样的东西，如何确保 AI 生成的代码经历与人类生成代码相同的审查流程？为了尽早应对这些问题，我们建议从一组已批准的 Skills、必需的代码审查流程和有限的初始访问权限开始，随着信心建立逐步扩展。

我们观察到，在那些能最顺利推广的组织中，他们通过组建跨职能工作组来早期介入——将工程、信息安全和合规治理代表聚集在一起，共同定义需求并制定推广路线图。

## 将这些模式应用于你的组织

Claude Code 围绕传统软件工程环境设计：工程师是代码库的主要贡献者，仓库使用 Git，代码遵循标准目录结构。大多数大型代码库符合这一模式，但非传统配置——例如拥有大型二进制资产的游戏引擎、使用非常规版本控制的环境，或非工程师向代码库贡献内容的场景——需要额外的配置工作。本文的指导假设一个传统的配置，我们描述的这些模式已在我们众多客户中得到验证。任何剩余的复杂性都需要针对你的代码库、工具链和组织的具体判断。这正是 Anthropic 应用 AI 团队与工程团队直接合作，将这些模式转化为你的组织特定需求的工作所在。
