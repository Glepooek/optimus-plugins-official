# 使用 CLAUDE.md 文件：为你的代码库定制 Claude Code

实用指南：如何使用 CLAUDE.md 文件优化 Claude Code 的使用效果。

> **来源：** [Claude Blog - Using CLAUDE.md files: Customizing Claude Code for your codebase](https://claude.com/blog/using-claude-md-files)
> **发布日期：** 2025 年 11 月 25 日
> **分类：** Claude Code | 产品：Claude Code | 阅读时长：5 分钟

---

如果你使用 AI 编程智能体，就会面临同样的挑战：如何在不重复自己的前提下，给它提供足够的上下文，让它理解你的架构、约定和工作流程？随着代码库的增长，这个问题会不断加剧。复杂的模块关系、领域特定模式和团队约定都不容易浮现。你最终会在每次对话开始时反复解释同样的架构决策、测试要求和代码风格偏好。

CLAUDE.md 文件通过为 Claude 提供关于项目的持久上下文来解决这个问题。把它想象成一个配置文件，Claude 会自动将其纳入每次对话，确保它始终了解你的项目结构、编码规范和首选工作流程。

本文将介绍如何构建你的 CLAUDE.md、分享最佳实践，以及如何利用它最大化 Claude Code 的效果。

---

## 一、什么是 CLAUDE.md 文件？

CLAUDE.md 是一个特殊的配置文件，存放在你的仓库中，为 Claude 提供项目特定的上下文。你可以将其放在仓库根目录以与团队共享，放在父目录用于 Monorepo 设置，或放在主目录下以跨所有项目通用。

以下是你可能在仓库中使用的 CLAUDE.md 示例：

```markdown
# 项目上下文

处理此代码库时，优先考虑可读性而非技巧性。在进行架构修改前先提出澄清问题。

## 关于此项目

用于用户认证和个人资料的 FastAPI REST API。使用 SQLAlchemy 进行数据库操作，使用 Pydantic 进行验证。

## 关键目录

- `app/models/` - 数据库模型
- `app/api/` - 路由处理器
- `app/core/` - 配置和工具

## 规范

- 所有函数必须有类型提示
- 使用 pytest 进行测试（fixtures 在 `tests/conftest.py`）
- PEP 8 规范，行长 100 字符

## 常用命令

```bash
uvicorn app.main:app --reload  # 开发服务器
pytest tests/ -v               # 运行测试
```

## 注意事项

所有路由使用 `/api/v1` 前缀。JWT token 24 小时后过期。
```

经过良好配置的 CLAUDE.md 能彻底改变 Claude 处理特定项目的方式。该文件服务于多个目的：提供架构上下文、建立工作流程，并将 Claude 与你的开发工具连接起来。每次新增内容都应该解决你实际遇到的问题，而非对 Claude 可能需要什么的理论预设。

此文件可以记录常用的 Bash 命令、核心工具库、代码风格指南、测试说明、仓库约定、开发环境设置以及项目特定警告。没有固定格式要求，建议保持简洁可读，将其视为人类和 Claude 都需要快速理解的文档。

**你的 CLAUDE.md 文件会成为 Claude 系统提示词的一部分。** 每次对话都从这些上下文已经加载的状态开始，消除了反复解释基本项目信息的需要。

## 二、使用 /init 快速入门

从头创建 CLAUDE.md 可能令人望而生畏，尤其是在不熟悉的代码库中。`/init` 命令通过分析你的项目并生成初始配置来自动化这一过程。

在任意 Claude Code 会话中运行 `/init`：

```bash
cd your-project
claude
/init
```

Claude 会检查你的代码库——读取包文件、现有文档、配置文件和代码结构——然后生成针对你项目的 CLAUDE.md。生成的文件通常包含构建命令、测试说明、关键目录以及它检测到的编码约定。

**把 `/init` 视为起点，而不是成品。** 生成的 CLAUDE.md 能捕捉明显的模式，但可能会遗漏你工作流程中的特定细节。检查 Claude 生成的内容并根据团队的实际实践加以完善。

你也可以在已有 CLAUDE.md 的现有项目上使用 `/init`，Claude 会审查当前文件并根据探索代码库所学到的内容提出改进建议。

运行 `/init` 后，建议完成以下步骤：
- 检查生成内容的准确性
- 补充 Claude 无法推断的工作流程说明（分支命名约定、部署流程、代码审查要求）
- 删除对你项目不适用的通用指导
- 将文件提交到版本控制，让团队受益

`/init` 命令适合快速上手，但真正的价值来自随时间对生成文件的迭代。在与 Claude Code 一起工作时，使用 `#` 键添加你发现自己反复说明的指令——这些积累会形成一个真正反映团队工作方式的 CLAUDE.md。

## 三、如何构建你的 CLAUDE.md

以下章节展示如何针对不同目的构建内容，以实现最大效果：导航复杂架构、追踪多步骤任务进度、集成自定义工具，以及通过一致的工作流程防止返工。

### 为 Claude 提供地图

在每次新任务中解释项目架构、关键库和编码风格是件繁琐的事。你需要 Claude 在不需要手动强化的情况下，对代码库结构保持一致的了解。

在 CLAUDE.md 中添加项目摘要和高层目录结构，让 Claude 在浏览代码库时能立即定位。一个简单的目录树展示关键目录，帮助 Claude 了解各组件的位置：

```
main.py
├── logs
│   ├── application.log
├── modules
│   ├── cli.py
│   ├── logging_utils.py
│   ├── media_handler.py
│   ├── player.py
```

包含你的主要依赖、架构模式以及任何非标准的组织选择。如果你使用领域驱动设计、微服务或特定框架，记录下来。Claude 会利用这张地图做出更好的决策，判断在哪里查找代码、在哪里进行修改。

### 将 Claude 连接到你的工具

Claude 继承了你的完整环境，但需要关于使用哪些自定义工具和脚本的指导。你的团队可能有用于部署、测试或代码生成的专用工具，Claude 应该了解它们。

在 CLAUDE.md 中记录你的自定义工具及使用示例，包括工具名称、基本使用模式以及何时调用它们。如果你的工具通过 `--help` 标志提供帮助文档，提一下这点，让 Claude 知道可以查阅。对于复杂工具，添加团队常用的调用示例。

Claude 作为 MCP（模型上下文协议）客户端运行，可以连接到扩展其能力的 MCP Server。通过项目设置、全局配置或已提交的 `.mcp.json` 文件进行配置。`--mcp-debug` 标志有助于在工具未按预期出现时排查连接问题。

例如，如果你为组织配置了 Slack MCP Server，并需要 Claude 了解如何使用它，可以在 CLAUDE.md 中添加：

```markdown
### Slack MCP
- 仅向 #dev-notifications 频道发布
- 用于部署通知和构建失败
- 不用于单个 PR 更新（这些通过 GitHub webhook 处理）
- 每小时限制 10 条消息
```

了解更多关于 [MCP 基础和最佳实践](https://modelcontextprotocol.io/)的信息。有关为 Claude Code 设置权限的更多信息，请参阅 [code.claude.com 的 settings.json 文档](https://code.claude.com/docs/en/settings)。

### 定义标准工作流程

让 Claude 不经规划就直接修改代码会造成返工。Claude 可能会实现一个不满足需求的解决方案、选择错误的架构方法，或做出破坏现有功能的改动。你需要 Claude 在行动前先思考。

在 CLAUDE.md 中定义针对不同任务类型的标准工作流程。一个好的默认工作流程在进行修改前先解决四个问题：
1. 这是否是一个需要先进行调查的关于当前状态的问题？
2. 这是否需要在实现前制定详细计划？
3. 还缺少哪些额外信息？
4. 如何测试有效性？

具体工作流程可能包括：针对功能的「探索-计划-编码-提交」流程，针对算法工作的测试驱动开发，或针对 UI 变更的视觉迭代。记录你的测试要求、提交消息格式以及任何审批步骤。

当 Claude 提前了解你的工作流程时，它会将工作结构化为符合团队实际流程的方式，而不是猜测。

工作流程指令示例：

```
1) 在修改以下位置的代码之前：X、Y、Z
   - 考虑可能对 A、B、C 产生的影响
   - 制定实现计划
   - 制定测试计划，用于验证以下函数……
```

## 四、与 Claude Code 协作的额外技巧

除了配置 CLAUDE.md 文件，三种额外的技术能改善你与 Claude Code 的协作效果。

### 保持上下文新鲜

长时间使用 Claude Code 会积累无关上下文。早期任务的文件内容、不再重要的命令输出以及旁支对话都会填满 Claude 的上下文窗口。随着信噪比下降，Claude 难以保持对当前任务的专注。

**使用 `/clear` 在不同任务之间重置上下文窗口。** 这会删除积累的历史记录，同时保留你的 CLAUDE.md 配置和 Claude 处理新问题的能力。把它理解为关闭一个工作会话、开启另一个。

当你完成认证调试并切换到实现新的 API 端点时，清除上下文。认证的细节已经不再重要，反而会干扰新工作。

### 为不同阶段使用子智能体

长对话积累的上下文会干扰新任务。你已经调试了复杂的认证流程，现在需要对同一段代码进行安全审查。调试细节会影响 Claude 的安全分析，可能导致它忽略问题或专注于已解决的问题。

**告诉 Claude 为工作的不同阶段使用子智能体。** 子智能体维护隔离的上下文，防止早期任务的信息干扰新的分析。实现支付处理器后，指示 Claude「使用子智能体对该代码进行安全审查」，而不是在同一对话中继续。

子智能体最适合多步骤工作流，每个阶段需要不同的视角。实现需要架构上下文和功能需求；安全审查需要专注于漏洞的新鲜视角。上下文分离让两种分析都保持敏锐。

### 创建自定义命令

重复的提示词浪费时间。你发现自己一遍遍输入「检查这段代码的安全问题」或「分析这段代码的性能问题」，每次都要记住能得到好结果的确切措辞。

**自定义斜杠命令**将这些提示词存储为 `.claude/commands/` 目录中的 Markdown 文件。创建名为 `performance-optimization.md` 的文件，包含你首选的性能优化提示词，它就会在任何对话中以 `/performance-optimization` 的形式可用。

命令通过 `$ARGUMENTS` 或 `$1`、`$2` 等带编号的占位符支持参数，让你可以传入特定文件或参数。

例如，`performance-optimization.md` 可能如下所示：

````markdown
# 性能优化

分析所提供代码的性能瓶颈和优化机会。进行全面审查，涵盖：

## 分析领域

### 数据库与数据访问
- N+1 查询问题和缺失的预加载
- 频繁查询列上缺少数据库索引
- 低效的关联查询或子查询
- 大结果集缺少分页
- 缺失查询结果缓存
- 连接池问题

### 算法效率
- 时间复杂度问题（O(n²) 或更差，但存在更好的方案）
- 可优化的嵌套循环
- 冗余计算或重复工作
- 低效的数据结构选择
- 缺失记忆化或动态规划机会

### 内存管理
- 内存泄漏或保留引用
- 应该使用流式处理时加载整个数据集
- 循环中过多对象实例化
- 不必要地将大型数据结构保留在内存中
- 缺失垃圾回收机会

### 异步与并发
- 应该异步的阻塞 I/O 操作
- 可以并行运行的顺序操作
- 缺失 Promise.all() 或并发执行模式
- 同步文件操作
- 未优化的工作线程使用

### 网络与 I/O
- 过多 API 调用（缺少请求批处理）
- 无响应缓存策略
- 没有压缩的大型负载
- 静态资源缺少 CDN
- 缺少连接复用

### 前端性能
- 阻塞渲染的 JavaScript 或 CSS
- 缺少代码分割或懒加载
- 未优化的图片或资源
- 过多 DOM 操作或回流
- 长列表缺少虚拟化
- 昂贵操作上没有防抖/节流

### 缓存
- 缺少 HTTP 缓存头
- 无应用层缓存
- 纯函数缺少记忆化
- 静态资源没有缓存破坏

## 输出格式

对于发现的每个问题：
1. **问题**：描述性能问题
2. **位置**：指定文件/函数/行号
3. **影响**：评定严重程度（Critical/High/Medium/Low）并解释预期的性能下降
4. **当前复杂度**：适用时包含时间/空间复杂度
5. **建议**：提供具体的优化策略
6. **代码示例**：可能时展示优化版本
7. **预期改进**：可衡量时量化性能提升

如果代码已经充分优化：
- 确认优化状态
- 列出正确实现的性能最佳实践
- 注意任何可能的小改进

**待审查代码：**
```
$ARGUMENTS
```
````

你不需要手动编写自定义命令文件。让 Claude 帮你创建：

```
创建一个名为 /performance-optimization 的自定义斜杠命令，用于分析代码的数据库查询问题、算法效率、内存管理和缓存机会。
```

Claude 会将 Markdown 文件写入 `.claude/commands/performance-optimization.md`，该命令会立即可用。

## 五、从简单开始，有意扩展

创建一份全面的 CLAUDE.md 的冲动是很正常的，但请克制它。CLAUDE.md 每次都会被添加到 Claude Code 的上下文中，因此从上下文工程和提示词工程的角度来看，**保持简洁**。

一种方案是：将信息拆分到独立的 Markdown 文件中，在 CLAUDE.md 文件内引用它们。

**不要包含敏感信息**：API 密钥、凭证、数据库连接字符串或详细的安全漏洞信息——尤其是在提交到版本控制时。由于 CLAUDE.md 成为 Claude 系统提示词的一部分，将其视为可以公开共享的文档。

## 六、让 CLAUDE.md 为你服务

CLAUDE.md 文件将 Claude Code 从一个通用助手变成专门为你的代码库配置的工具。从基本的项目结构和构建文档开始，然后根据工作流中实际的摩擦点进行扩展。

最有效的 CLAUDE.md 文件解决真实问题：记录你反复输入的命令，捕捉需要十分钟解释的架构上下文，以及建立防止返工的工作流程。你的文件应该反映团队实际开发软件的方式——而不是听起来不错但与现实不符的理论最佳实践。

将定制化视为一种持续实践，而非一次性设置任务。项目会变化，团队会学到更好的模式，新工具会进入工作流程。维护良好的 CLAUDE.md 随着代码库一同演进，持续减少使用 AI 辅助处理复杂软件的摩擦。
